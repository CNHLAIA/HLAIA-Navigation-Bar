## 1. 前置知识

在正式进入书签模块之前，我们需要先理解几个数据库和 Java 概念。如果你对 SQL 基础已经比较熟，可以快速浏览这一节。

### 1.1 SQL 分页：LIMIT 和 OFFSET

假设你的书签表里有 10000 条记录。如果执行：

```sql
SELECT * FROM bookmark WHERE user_id = 1;
```

数据库会把 10000 条记录**全部**返回。这对前端和后端都是灾难——前端渲染 10000 个书签会卡顿，后端传输这么多数据会占用大量带宽。

**分页**就是把数据"切成一页一页"地取。就像翻书一样，每次只看一页（比如 20 条），而不是一次看完所有内容。

MySQL 用 `LIMIT` 和 `OFFSET` 实现分页：

```sql
-- 第 1 页：每页 20 条，从第 0 条开始
SELECT * FROM bookmark WHERE user_id = 1
ORDER BY sort_order ASC
LIMIT 20 OFFSET 0;

-- 第 2 页：每页 20 条，从第 20 条开始
SELECT * FROM bookmark WHERE user_id = 1
ORDER BY sort_order ASC
LIMIT 20 OFFSET 20;

-- 第 3 页：每页 20 条，从第 40 条开始
SELECT * FROM bookmark WHERE user_id = 1
ORDER BY sort_order ASC
LIMIT 20 OFFSET 40;
```

记住这个公式：**OFFSET = (页码 - 1) x 每页条数**。

- `LIMIT 20` 表示最多返回 20 条记录
- `OFFSET 0` 表示从第 0 条开始（MySQL 的下标从 0 开始计数）
- 第 2 页就是 `OFFSET 20`，跳过前 20 条，从第 21 条开始取

还有一种简写方式：`LIMIT offset, count`，比如 `LIMIT 0, 20` 等价于 `LIMIT 20 OFFSET 0`。

### 1.2 SQL 批量操作：IN 语句

如果要删除 ID 为 1、3、5 的三条记录，你当然可以执行三次 DELETE：

```sql
DELETE FROM bookmark WHERE id = 1;
DELETE FROM bookmark WHERE id = 3;
DELETE FROM bookmark WHERE id = 5;
```

但这样效率低——每次都要和数据库通信一次。**IN 语句**可以让一条 SQL 搞定：

```sql
DELETE FROM bookmark WHERE id IN (1, 3, 5);
```

`IN` 的含义是"在这个列表里"。数据库只需要解析和执行一次 SQL，性能好很多。

Java 中的对应写法：MyBatis-Plus 提供了 `deleteBatchIds` 方法，底层就是生成 `DELETE FROM 表 WHERE id IN (?, ?, ?)` 这样的 SQL。

### 1.3 外键关系：书签属于文件夹

在之前学习文件夹（Folder）的时候，我们知道文件夹用 `parent_id` 字段形成树形结构。书签（Bookmark）和文件夹之间是**多对一**的关系：

```
文件夹（1）─── 包含 ───→ （多）书签
```

一个文件夹可以包含多个书签，但一个书签只能属于一个文件夹。

数据库中通过 `folder_id` 字段来实现这种关系：

```
bookmark 表                        folder 表
+----+---------+-----------+       +----+---------+
| id | user_id | folder_id |       | id | user_id |
+----+---------+-----------+       +----+---------+
|  1 |       1 |         2 | ───→  |  2 |       1 |  "前端工具"
|  2 |       1 |         2 | ───→  |  2 |       1 |  "前端工具"
|  3 |       1 |         3 | ───→  |  3 |       1 |  "后端工具"
+----+---------+-----------+       +----+---------+
```

`bookmark.folder_id` 的值等于 `folder.id`，就像你的身份证上写着"家庭地址"一样——它告诉你这个书签"住"在哪个文件夹里。

### 1.4 DTO（数据传输对象）回顾

在前面的阶段中你已经接触过 DTO。这里再快速回顾一下它的核心思想：

- **Entity（实体类）**：对应数据库表结构，包含所有字段。比如 `Bookmark` 有 `userId` 字段。
- **DTO（数据传输对象）**：只包含前端需要的数据。比如 `BookmarkResponse` 没有 `userId`（前端不需要知道用户 ID）。

为什么不直接返回 Entity？三个原因：

1. **安全性**：Entity 可能包含敏感字段（如 userId），不应暴露给前端
2. **解耦**：数据库结构变化不会直接影响 API 接口
3. **灵活性**：DTO 可以组合多个 Entity 的字段，或添加计算字段

## 2. 概念讲解

### 2.1 分页查询：为什么不能一次查所有数据？

你可能想问："我的书签总共就几十个，为什么要分页？"

现在的数据量确实不大。但在真实的商业项目中，数据量可能是百万甚至千万级别的。不分页的后果：

- **内存溢出**：把 100 万条记录全部加载到 Java 内存中，可能直接导致 OOM（Out Of Memory）
- **响应缓慢**：网络传输大量数据需要很长时间
- **前端卡顿**：浏览器渲染 10 万个 DOM 元素会变得非常卡

所以，**从一开始就养成"分页查询"的习惯**是很好的做法，即使现在数据量不大。

### 2.2 MyBatis-Plus 分页插件：自动给 SQL 加 LIMIT

MyBatis-Plus 提供了一个非常方便的分页插件。你只需要告诉它"第几页、每页几条"，它就会自动帮你：

1. 在原始 SQL 后面加上 `LIMIT ... OFFSET ...`
2. 自动执行一条 `SELECT COUNT(*)` 来获取总记录数
3. 把结果封装到 `Page<T>` 对象中

使用分页插件的代码模式：

```java
// 第一步：创建 Page 对象（告诉 MyBatis-Plus 你要第几页、每页几条）
Page<Bookmark> page = new Page<>(1, 20);   // 第 1 页，每页 20 条

// 第二步：创建查询条件（和普通查询一样）
LambdaQueryWrapper<Bookmark> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(Bookmark::getUserId, userId);

// 第三步：执行分页查询（自动加上 LIMIT）
Page<Bookmark> result = bookmarkMapper.selectPage(page, wrapper);

// 第四步：从 result 中获取数据
List<Bookmark> records = result.getRecords();   // 当前页的数据列表
long total = result.getTotal();                  // 总记录数
long pages = result.getPages();                  // 总页数
```

但这有个前提——**必须先配置分页插件**，否则 `selectPage` 不会自动加 LIMIT，会返回所有数据。

### 2.3 分页插件配置：MyBatisPlusConfig

打开项目中的 `MyBatisPlusConfig.java`，你会看到这样的配置：

```java
@Configuration
public class MyBatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        // 添加分页插件，指定数据库类型为 MySQL
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
        return interceptor;
    }
}
```

这个配置做了什么？

1. `@Configuration` 告诉 Spring：这是一个配置类
2. `@Bean` 告诉 Spring：这个方法返回的对象交给 Spring 容器管理
3. `MybatisPlusInterceptor` 是 MyBatis-Plus 的拦截器（你可以理解为"插件管理器"）
4. `PaginationInnerInterceptor(DbType.MYSQL)` 是分页插件本体，`DbType.MYSQL` 告诉它生成 MySQL 语法的分页 SQL

**工作原理**：当 MyBatis-Plus 执行 `selectPage` 方法时，分页拦截器会：

1. 拦截原始 SQL：`SELECT * FROM bookmark WHERE user_id = ?`
2. 改写为分页 SQL：`SELECT * FROM bookmark WHERE user_id = ? LIMIT 20 OFFSET 0`
3. 额外执行一条 COUNT SQL：`SELECT COUNT(*) FROM bookmark WHERE user_id = ?`
4. 把两条 SQL 的结果封装到 `Page<Bookmark>` 对象中

这就是为什么**必须配置分页插件**——没有它，拦截器不会生效，`selectPage` 就只是普通查询。

> **本项目说明**：当前项目中书签的查询接口 `getBookmarksByFolder` 使用的是 `selectList`（查询全部）而不是 `selectPage`（分页查询）。这是因为一个文件夹下的书签数量通常不会太多，不需要分页。但分页插件的配置已经就绪，如果未来某个接口需要分页，直接使用 `selectPage` 即可。本项目在文件夹管理等可能涉及大量数据的场景中已预留了分页支持。

### 2.4 批量操作：一次处理多条数据

批量操作是指在一个请求中同时处理多条记录。本项目的书签模块有三种批量操作：

#### 批量排序（sortBookmarks）

用户拖拽书签改变顺序后，前端发送一个排序列表：

```json
{
  "items": [
    { "id": 1, "sortOrder": 0 },
    { "id": 2, "sortOrder": 1 },
    { "id": 3, "sortOrder": 2 }
  ]
}
```

后端遍历这个列表，逐个更新每条记录的 `sortOrder`。

#### 批量删除（batchDelete）

前端发送一组 ID：

```json
{
  "ids": [1, 3, 5]
}
```

后端先逐个校验权限，再一次性删除。底层生成 SQL：`DELETE FROM bookmark WHERE id IN (1, 3, 5)`。

#### 批量复制链接（batchCopyLinks）

前端发送一组 ID，后端返回这些书签的 URL 列表：

```json
// 请求
{ "ids": [1, 2, 3] }

// 响应
["https://www.baidu.com", "https://github.com", "https://stackoverflow.com"]
```

### 2.5 Entity 到 Response 的映射

在书签模块中，每次查询到 `Bookmark` 实体后，都需要转换成 `BookmarkResponse` 再返回给前端。这个过程叫做 **DTO 映射**。

```
┌─────────────────┐          ┌─────────────────────┐
│   Bookmark       │          │  BookmarkResponse    │
│   (Entity)       │  映射    │  (DTO)               │
│                  │ ──────→  │                      │
│ id               │          │ id                   │
│ userId      ← 不传 │          │ folderId             │
│ folderId         │          │ title                │
│ title            │          │ url                  │
│ url              │          │ description          │
│ description      │          │ iconUrl              │
│ iconUrl          │          │ sortOrder            │
│ sortOrder        │          │ createdAt            │
│ createdAt        │          │ updatedAt            │
│ updatedAt        │          │                      │
└─────────────────┘          └─────────────────────┘
```

注意 `userId` 没有出现在 Response 中——前端不需要知道书签属于哪个用户，因为用户只能看到自己的书签。

## 3. 代码逐行解读

现在我们按照数据流向，从 Entity 开始，逐层解读书签模块的完整代码。

### 3.1 Entity 层：Bookmark.java

> 文件路径：`src/main/java/com/hlaia/entity/Bookmark.java`

```java
@Data
@TableName("bookmark")
public class Bookmark {

    @TableId(type = IdType.AUTO)
    private Long id;

    private Long userId;       // 所属用户
    private Long folderId;     // 所属文件夹（外键）
    private String title;      // 书签标题
    private String url;        // 链接地址
    private String description;// 描述（可选）
    private String iconUrl;    // 网站图标 URL
    private Integer sortOrder; // 排序序号

    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
}
```

逐个字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | Long | 主键，`@TableId(type = IdType.AUTO)` 表示数据库自增 |
| `userId` | Long | 外键，关联 user 表。每个用户只能操作自己的书签 |
| `folderId` | Long | 外键，关联 folder 表。书签必须属于某个文件夹 |
| `title` | String | 显示在导航栏上的文字，如 "百度" |
| `url` | String | 链接地址，如 "https://www.baidu.com"。数据库有唯一索引 (user_id, url) |
| `description` | String | 备注说明，可以为 null |
| `iconUrl` | String | 网站小图标地址，可以为 null（创建后再异步获取） |
| `sortOrder` | Integer | 排序值，同一个文件夹内按此字段升序排列 |

与 Folder 实体的对比：

- Folder 用 `parentId` 实现树形结构（一个文件夹可以有子文件夹）
- Bookmark 用 `folderId` 表示归属（一个书签只属于一个文件夹）
- Bookmark 多了 `url` 和 `iconUrl` 字段（书签的核心就是链接）

### 3.2 Mapper 层：BookmarkMapper.java

> 文件路径：`src/main/java/com/hlaia/mapper/BookmarkMapper.java`

```java
@Mapper
public interface BookmarkMapper extends BaseMapper<Bookmark> {
}
```

和之前的 FolderMapper 一样，继承 `BaseMapper<Bookmark>` 后就自动拥有了基础的 CRUD 方法：

| 方法 | 作用 | 生成的 SQL |
|------|------|-----------|
| `selectById(id)` | 按主键查询 | `SELECT * FROM bookmark WHERE id = ?` |
| `selectList(wrapper)` | 按条件查询列表 | `SELECT * FROM bookmark WHERE ... ORDER BY ...` |
| `selectPage(page, wrapper)` | 分页查询 | `SELECT * FROM bookmark WHERE ... LIMIT ? OFFSET ?` |
| `selectCount(wrapper)` | 统计数量 | `SELECT COUNT(*) FROM bookmark WHERE ...` |
| `insert(entity)` | 插入一条记录 | `INSERT INTO bookmark (...) VALUES (...)` |
| `updateById(entity)` | 按主键更新 | `UPDATE bookmark SET ... WHERE id = ?` |
| `deleteById(id)` | 按主键删除 | `DELETE FROM bookmark WHERE id = ?` |
| `deleteBatchIds(ids)` | 批量删除 | `DELETE FROM bookmark WHERE id IN (?, ?, ?)` |

你不需要自己写这些 SQL——MyBatis-Plus 已经帮你生成了。如果将来需要复杂查询（比如多表联查），可以在接口中添加自定义方法并写 XML 映射文件。

### 3.3 DTO 层：请求和响应对象

书签模块有三个请求 DTO 和一个响应 DTO。

#### BookmarkCreateRequest.java（创建书签请求）

```java
@Data
public class BookmarkCreateRequest {

    @NotNull(message = "文件夹ID不能为空")
    private Long folderId;       // 必填：书签必须放在某个文件夹里

    @NotBlank(message = "书签标题不能为空")
    private String title;        // 必填：书签需要一个名字

    @NotBlank(message = "书签链接不能为空")
    private String url;          // 必填：链接是书签的核心

    private String description;  // 可选：用户可以不填
}
```

**校验注解的选择**：

- `Long` 类型用 `@NotNull`（不能为 null）
- `String` 类型用 `@NotBlank`（不能为 null、空字符串 "" 或纯空格 "   "）
- 可选字段不加校验注解（前端不传时自动为 null）

注意 `@NotBlank` 只能用在 `String` 类型上。如果对 `Long` 类型用 `@NotBlank`，启动时会报类型不匹配的错误。这是一个初学者常见的坑。

#### BookmarkSortRequest.java（排序请求）

```java
@Data
public class BookmarkSortRequest {

    @NotNull(message = "排序列表不能为空")
    @Valid   // 嵌套校验：校验 SortItem 内部的字段
    private List<SortItem> items;

    @Data
    public static class SortItem {    // 静态内部类
        @NotNull(message = "书签ID不能为空")
        private Long id;

        @NotNull(message = "排序序号不能为空")
        private Integer sortOrder;
    }
}
```

这里出现了一个新概念——**静态内部类**（`static class`）。

`SortItem` 定义在 `BookmarkSortRequest` 的内部，因为排序项只和书签排序有关，不需要被其他类使用。用 `static` 修饰意味着：

- 不需要先创建外部类的实例就能使用 `SortItem`
- `SortItem` 不能访问外部类的实例字段（其实外部类也没实例字段）

这是一种"高内聚"的组织方式——相关的类放在一起，便于理解和维护。

`@Valid` 注解也很关键。它告诉 Spring：不仅要校验 `items` 不为 null，还要**深入到列表内部**，校验每个 `SortItem` 的 `id` 和 `sortOrder`。不加 `@Valid` 的话，`SortItem` 里的 `@NotNull` 注解不会生效。

#### BatchDeleteRequest.java（批量删除请求）

```java
@Data
public class BatchDeleteRequest {

    @NotEmpty(message = "删除列表不能为空")
    private List<Long> ids;
}
```

注意这里用的是 `@NotEmpty` 而不是 `@NotNull`：

| 注解 | null | 空列表 [] | 空字符串 "" |
|------|------|-----------|------------|
| `@NotNull` | 不允许 | 允许 | 允许 |
| `@NotEmpty` | 不允许 | 不允许 | 不允许 |
| `@NotBlank` | 不允许 | 不允许 | 不允许（还要求不能全是空格） |

"删除 0 个东西"没有意义，所以用 `@NotEmpty` 确保列表至少有一个元素。

#### BookmarkResponse.java（书签响应）

```java
@Data
public class BookmarkResponse {
    private Long id;
    private Long folderId;
    private String title;
    private String url;
    private String description;
    private String iconUrl;
    private Integer sortOrder;
```

和 `Bookmark` 实体对比，最大的区别是**没有 `userId` 字段**。

为什么？因为前端不需要这个信息。用户登录后只能看到自己的书签，不需要前端知道"这是用户 ID 为 1 的书签"。隐藏这类内部字段是 DTO 存在的重要意义之一。

### 3.4 Service 层：BookmarkService.java

> 文件路径：`src/main/java/com/hlaia/service/BookmarkService.java`

这是书签模块的核心，我们逐个方法解读。

#### getBookmarksByFolder()：按文件夹查询书签

```java
public List<BookmarkResponse> getBookmarksByFolder(Long userId, Long folderId) {
    List<Bookmark> bookmarks = bookmarkMapper.selectList(
            new LambdaQueryWrapper<Bookmark>()
                    .eq(Bookmark::getUserId, userId)
                    .eq(Bookmark::getFolderId, folderId)
                    .orderByAsc(Bookmark::getSortOrder));

    return bookmarks.stream()
            .map(this::toResponse)
            .collect(Collectors.toList());
}
```

这段代码的执行流程：

1. **构建查询条件**：`LambdaQueryWrapper` 是 MyBatis-Plus 的查询构造器
   - `.eq(Bookmark::getUserId, userId)` → SQL: `WHERE user_id = ?`
   - `.eq(Bookmark::getFolderId, folderId)` → SQL: `AND folder_id = ?`
   - `.orderByAsc(Bookmark::getSortOrder)` → SQL: `ORDER BY sort_order ASC`

2. **执行查询**：`bookmarkMapper.selectList(wrapper)` 生成最终 SQL：
   ```sql
   SELECT * FROM bookmark WHERE user_id = ? AND folder_id = ? ORDER BY sort_order ASC
   ```

3. **转换结果**：用 Stream API 把 `List<Bookmark>` 转成 `List<BookmarkResponse>`
   - `stream()` 把列表转成流
   - `map(this::toResponse)` 对每个 Bookmark 调用 `toResponse` 方法
   - `collect(Collectors.toList())` 把流转回 List

> **如果将来需要分页**，只需要把 `selectList` 改成 `selectPage`：
> ```java
> Page<Bookmark> page = new Page<>(pageNum, pageSize);
> Page<Bookmark> result = bookmarkMapper.selectPage(page, wrapper);
> List<Bookmark> bookmarks = result.getRecords();  // 当前页数据
> long total = result.getTotal();                    // 总记录数
> ```
> 分页插件会自动在 SQL 后面加上 `LIMIT` 和 `OFFSET`。

#### createBookmark()：创建书签

```java
@Transactional
public BookmarkResponse createBookmark(Long userId, BookmarkCreateRequest request) {
    // 第一步：构建 Entity 对象
    Bookmark bookmark = new Bookmark();
    bookmark.setUserId(userId);
    bookmark.setFolderId(request.getFolderId());
    bookmark.setTitle(request.getTitle());
    bookmark.setUrl(request.getUrl());
    bookmark.setDescription(request.getDescription());

    // 第二步：计算排序序号（新书签排在最后面）
    Long count = bookmarkMapper.selectCount(
            new LambdaQueryWrapper<Bookmark>()
                    .eq(Bookmark::getFolderId, request.getFolderId()));
    bookmark.setSortOrder(count.intValue());

    // 第三步：插入数据库
    bookmarkMapper.insert(bookmark);

    // 第四步：发送 Kafka 异步任务（获取网站图标）
    kafkaProducer.sendIconFetchTask(bookmark.getId(), bookmark.getUrl());

    return toResponse(bookmark);
}
```

这个方法的四个步骤值得仔细理解：

**第一步：DTO → Entity 转换**

前端传过来的是 `BookmarkCreateRequest`（DTO），但数据库需要的是 `Bookmark`（Entity）。所以我们需要手动把 DTO 的字段"搬"到 Entity 上。

你可能会想："为什么不直接用 Entity 接收前端数据？"因为 DTO 和 Entity 的职责不同：

- DTO 是给前端用的，只包含前端应该提交的字段
- Entity 是给数据库用的，包含所有字段（包括 `userId`、`createdAt` 等前端不应该设置的字段）

**第二步：计算排序序号**

先查询这个文件夹下已有多少书签，新书签的 `sortOrder` 就等于已有数量。这样新书签自然排在最后。

```java
Long count = bookmarkMapper.selectCount(...);  // 查数量
bookmark.setSortOrder(count.intValue());        // Long → int
```

`count.intValue()` 是把 `Long`（包装类）转成 `int`（基本类型）。MyBatis-Plus 的 `selectCount` 返回 `Long`，而 `sortOrder` 字段是 `Integer`，所以需要转换。

**第三步：插入数据库**

`bookmarkMapper.insert(bookmark)` 会生成并执行：

```sql
INSERT INTO bookmark (user_id, folder_id, title, url, description, sort_order)
VALUES (?, ?, ?, ?, ?, ?)
```

插入成功后，`bookmark.getId()` 会自动获得数据库生成的自增主键值。这是 MyBatis-Plus 的默认行为——插入后自动回填主键。

**第四步：异步获取图标**

```java
kafkaProducer.sendIconFetchTask(bookmark.getId(), bookmark.getUrl());
```

用户创建书签后，后端需要访问该网站来获取 favicon（浏览器标签页上的小图标）。这个过程可能需要几秒钟（网络请求），如果同步获取会拖慢接口响应。

所以这里使用 Kafka 异步处理：

1. 创建书签时，只发送一条消息到 Kafka
2. Kafka 消费者在后台获取图标，成功后更新 `iconUrl` 字段
3. 前端下次刷新就能看到图标了

这种"让用户先看到结果，后台慢慢处理"的设计叫做**异步处理**，是提升用户体验的常见手段。

**@Transactional 注解**

`@Transactional` 保证方法内的所有数据库操作要么全部成功，要么全部失败。这里涉及 `selectCount` + `insert` 两次数据库操作，需要事务保护。

注意：Kafka 发送是异步的，不在事务范围内。即使 Kafka 发送失败，书签也已经创建成功了，只是没有图标而已。

#### updateBookmark()：更新书签

```java
@Transactional
public BookmarkResponse updateBookmark(Long userId, Long bookmarkId,
                                        BookmarkCreateRequest request) {
    Bookmark bookmark = getBookmarkForUser(userId, bookmarkId);

    if (request.getTitle() != null) bookmark.setTitle(request.getTitle());
    if (request.getUrl() != null) bookmark.setUrl(request.getUrl());
    if (request.getDescription() != null) bookmark.setDescription(request.getDescription());

    bookmarkMapper.updateById(bookmark);
    return toResponse(bookmark);
}
```

这里采用了**部分更新策略**：只有 request 中非 null 的字段才会被更新。

比如用户只想改标题，前端可以只传：

```json
{ "title": "新的标题" }
```

`url` 和 `description` 为 null，if 条件不成立，这两个字段不会被修改。这种设计让前端非常灵活——不需要每次都传所有字段。

#### deleteBookmark()：删除单个书签

```java
@Transactional
public void deleteBookmark(Long userId, Long bookmarkId) {
    getBookmarkForUser(userId, bookmarkId);
    bookmarkMapper.deleteById(bookmarkId);
}
```

两步操作：

1. 权限校验：确保书签存在且属于当前用户
2. 删除：`deleteById` 生成 `DELETE FROM bookmark WHERE id = ?`

这里使用的是**物理删除**——直接从数据库中移除记录。在生产环境中，通常会使用**逻辑删除**（设置 `is_deleted = 1`），这样数据还能恢复。但本项目作为学习项目，使用物理删除更简单直观。

返回值是 `void`——删除成功后资源已不存在，没有需要返回的数据。

#### sortBookmarks()：批量排序

```java
@Transactional
public void sortBookmarks(Long userId, BookmarkSortRequest request) {
    for (BookmarkSortRequest.SortItem item : request.getItems()) {
        Bookmark bookmark = getBookmarkForUser(userId, item.getId());
        bookmark.setSortOrder(item.getSortOrder());
        bookmarkMapper.updateById(bookmark);
    }
}
```

拖拽排序的实现原理：

1. 用户在前端拖拽书签改变顺序
2. 前端重新计算所有受影响书签的 `sortOrder`，发送到后端
3. 后端遍历列表，逐个更新

每次更新前都调用 `getBookmarkForUser` 做权限校验——防止恶意用户把别人的书签排序搞乱。

`@Transactional` 在这里特别重要：如果更新到一半失败了（比如第 3 个书签不存在），前面已经更新的 2 个需要回滚，否则排序数据就不一致了。

#### batchDelete()：批量删除

```java
@Transactional
public void batchDelete(Long userId, BatchDeleteRequest request) {
    // 第一步：逐个校验所有权
    for (Long id : request.getIds()) {
        getBookmarkForUser(userId, id);
    }
    // 第二步：批量删除
    bookmarkMapper.deleteBatchIds(request.getIds());
}
```

这个方法采用了**"先校验再删除"**的模式，而不是"边删边校验"。为什么？

考虑这个场景：用户要删除 ID 为 [1, 2, 3] 的书签，但 ID=3 属于另一个用户。

- 如果"边删边校验"：先删 1（成功），再删 2（成功），到 3 时发现无权限抛异常。但 1 和 2 已经被删了！
- "先校验再删除"：先检查 1（通过）、2（通过）、3（失败抛异常）。因为 `@Transactional`，整个方法回滚，什么都没删。

"先校验再删除"保证了**原子性**——要么全删，要么一个都不删。

`deleteBatchIds` 底层生成的 SQL：

```sql
DELETE FROM bookmark WHERE id IN (?, ?, ?)
```

只需要一条 SQL 就能删除多条记录，比循环调用 `deleteById` 效率高得多。

#### batchCopyLinks()：批量复制链接

```java
public List<String> batchCopyLinks(Long userId, BatchCopyRequest request) {
    return request.getIds().stream()
            .map(id -> {
                Bookmark b = getBookmarkForUser(userId, id);
                return b.getUrl();
            })
            .collect(Collectors.toList());
}
```

这个方法返回 `List<String>`（URL 列表），而不是 `List<BookmarkResponse>`。因为前端只需要 URL 字符串来复制到剪贴板，不需要标题、图标等其他信息。

返回精简数据可以减少网络传输量，也简化了前端处理逻辑。

这里用到了 Stream API，我们在第 4 节会详细讲解。

#### getBookmarkForUser()：权限校验辅助方法

```java
private Bookmark getBookmarkForUser(Long userId, Long bookmarkId) {
    Bookmark bookmark = bookmarkMapper.selectById(bookmarkId);
    if (bookmark == null || !bookmark.getUserId().equals(userId)) {
        throw new BusinessException(ErrorCode.BOOKMARK_NOT_FOUND);
    }
    return bookmark;
}
```

这个私有方法在 Service 内部被多个方法复用。它做了两件事：

1. 查询书签是否存在
2. 校验书签是否属于当前用户

如果书签不存在或者不属于当前用户，都抛出"书签不存在"异常——而不是"无权限"。这是安全最佳实践，避免泄露信息（攻击者无法判断资源是否存在）。

#### toResponse()：Entity → DTO 转换

```java
private BookmarkResponse toResponse(Bookmark bookmark) {
    BookmarkResponse dto = new BookmarkResponse();
    dto.setId(bookmark.getId());
    dto.setFolderId(bookmark.getFolderId());
    dto.setTitle(bookmark.getTitle());
    dto.setUrl(bookmark.getUrl());
    dto.setDescription(bookmark.getDescription());
    dto.setIconUrl(bookmark.getIconUrl());
    dto.setSortOrder(bookmark.getSortOrder());
    dto.setCreatedAt(bookmark.getCreatedAt());
    dto.setUpdatedAt(bookmark.getUpdatedAt());
    return dto;
}
```

这个方法做的事情很简单：把 `Bookmark` 的字段逐个"搬"到 `BookmarkResponse` 中。注意 `userId` 没有被搬运——这就是 DTO 过滤敏感字段的具体实现。

在实际商业项目中，这种映射可以用 **MapStruct** 等工具自动生成，不需要手写。但学习阶段手写有助于理解底层原理。

### 3.5 Controller 层：BookmarkController.java

> 文件路径：`src/main/java/com/hlaia/controller/BookmarkController.java`

Controller 层负责接收 HTTP 请求，调用 Service，返回结果。

#### 嵌套资源路径设计

获取书签列表的接口使用了嵌套路径：

```
GET /api/folders/{folderId}/bookmarks
```

这种 URL 结构清晰地表达了"文件夹包含书签"的从属关系。对比平铺路径：

```
GET /api/bookmarks?folderId=3
```

嵌套路径语义更直观，一看就知道是"文件夹 3 下的书签"。这是 RESTful API 设计中"资源嵌套"的常见做法。

#### RESTful API 一览

| HTTP 方法 | 路径 | 说明 |
|-----------|------|------|
| GET | `/api/folders/{folderId}/bookmarks` | 获取文件夹下的书签 |
| POST | `/api/bookmarks` | 创建书签 |
| PUT | `/api/bookmarks/{id}` | 更新书签 |
| DELETE | `/api/bookmarks/{id}` | 删除书签 |
| PUT | `/api/bookmarks/sort` | 批量排序 |
| POST | `/api/bookmarks/batch-delete` | 批量删除 |
| POST | `/api/bookmarks/batch-copy` | 批量复制链接 |

#### 批量删除为什么用 POST 而不是 DELETE？

按照 REST 纯粹主义，删除应该用 DELETE 方法。但批量删除有个问题：

- DELETE 方法通常不携带请求体（HTTP 规范不禁止，但很多客户端和代理不支持）
- 批量操作需要传递 ID 列表，只能放在请求体中
- 所以批量删除使用 POST + `/batch-delete` 路径，这是业界标准做法

#### @AuthenticationPrincipal 的作用

```java
@GetMapping("/folders/{folderId}/bookmarks")
public Result<List<BookmarkResponse>> list(
        @AuthenticationPrincipal Long userId,
        @PathVariable Long folderId) {
    return Result.success(bookmarkService.getBookmarksByFolder(userId, folderId));
}
```

`@AuthenticationPrincipal` 可以直接拿到当前登录用户的 ID。这个值是在 JWT 认证过滤器中设置的：

```java
// JwtAuthFilter 中设置 principal
UsernamePasswordAuthenticationToken auth =
    new UsernamePasswordAuthenticationToken(userId, null, authorities);
```

所以 Controller 方法不需要手动从 Token 中解析用户 ID——Spring Security 自动注入了。

#### @Valid 触发校验

```java
@PostMapping("/bookmarks")
public Result<BookmarkResponse> create(
        @AuthenticationPrincipal Long userId,
        @Valid @RequestBody BookmarkCreateRequest request) {
    return Result.success(bookmarkService.createBookmark(userId, request));
}
```

`@Valid` 会触发 `BookmarkCreateRequest` 字段上的校验注解（`@NotNull`、`@NotBlank`）。如果不加 `@Valid`，这些注解形同虚设。

校验失败时，Spring 自动抛出 `MethodArgumentNotValidException`，被全局异常处理器捕获并返回 400 错误。

## 4. 关键 Java 语法点

### 4.1 Page 对象和泛型

MyBatis-Plus 的分页查询返回 `Page<T>` 对象，这里的 `T` 是泛型参数。

```java
// T = Bookmark，表示这一页里存的是 Bookmark 对象
Page<Bookmark> page = new Page<>(1, 20);

// 执行分页查询
Page<Bookmark> result = bookmarkMapper.selectPage(page, wrapper);

// 从分页结果中获取数据
List<Bookmark> records = result.getRecords();  // 当前页的数据
long total = result.getTotal();                 // 总记录数
long pages = result.getPages();                 // 总页数 = ceil(total / pageSize)
long current = result.getCurrent();             // 当前页码
```

泛型的好处：`Page<Bookmark>` 和 `Page<Folder>` 是不同的类型，编译器会帮你检查类型安全。如果你不小心把 `Page<Bookmark>` 当成 `Page<Folder>` 用，编译时就会报错。

### 4.2 Stream API 的 map 操作

Stream API 是 Java 8 引入的函数式编程特性。在书签模块中，`map` 操作用于把一种类型的列表转换成另一种类型的列表：

```java
// 将 List<Bookmark> 转换为 List<BookmarkResponse>
List<BookmarkResponse> result = bookmarks.stream()
        .map(this::toResponse)
        .collect(Collectors.toList());
```

类比理解：

```
原始列表：  [Bookmark1, Bookmark2, Bookmark3]
                ↓ map      ↓ map      ↓ map
转换后：    [Response1, Response2, Response3]
```

`map` 的作用就是对列表中的**每个元素**执行同一个转换操作。

`this::toResponse` 是**方法引用**，等价于 `bookmark -> this.toResponse(bookmark)`。意思是：对每个 bookmark，调用 `this.toResponse(bookmark)` 方法。

Stream 操作的三步曲：

1. `stream()` —— 把集合变成流（打开水龙头）
2. `map(...)` —— 中间操作，对流中的每个元素做变换（过滤器）
3. `collect(Collectors.toList())` —— 终止操作，把流转回集合（关水龙头接水）

### 4.3 Set 和 List 的批量操作

在批量操作中，我们经常使用 `List<Long>` 来传递一组 ID：

```java
// BatchDeleteRequest 中的 ids 字段
private List<Long> ids;
```

`List` 允许重复元素，且有序。如果要判断某个 ID 是否在列表中，`List.contains()` 的时间复杂度是 O(n)（需要遍历）。

如果需要频繁判断"是否包含"，应该使用 `Set`（时间复杂度 O(1)）：

```java
List<Long> idList = Arrays.asList(1L, 3L, 5L);
Set<Long> idSet = new HashSet<>(idList);   // List → Set

idSet.contains(3L);  // true，O(1) 时间
idList.contains(3L); // true，O(n) 时间
```

在本项目的批量删除中，我们直接把 `List<Long>` 传给 `deleteBatchIds`，因为 MyBatis-Plus 底层会生成 `WHERE id IN (?, ?, ?)`，不需要我们手动判断包含关系。

### 4.4 @RequestParam：查询参数绑定

虽然本项目的书签列表接口使用了路径参数（`@PathVariable`），但在很多分页查询场景中，分页参数是通过 URL 查询字符串传递的：

```
GET /api/bookmarks?folderId=3&page=1&size=20
```

对应的 Controller 方法可以这样写：

```java
@GetMapping("/bookmarks")
public Result<Page<BookmarkResponse>> list(
        @RequestParam(defaultValue = "1") int page,
        @RequestParam(defaultValue = "20") int size,
        @RequestParam Long folderId) {
    // ...
}
```

`@RequestParam` 的作用是把 URL 中 `?` 后面的参数绑定到方法参数上。`defaultValue` 设置默认值——如果前端不传 `page`，默认就是第 1 页。

三种参数绑定注解的区别：

| 注解 | 数据来源 | 示例 |
|------|---------|------|
| `@PathVariable` | URL 路径 | `/bookmarks/{id}` → id=42 |
| `@RequestParam` | URL 查询字符串 | `?page=1&size=20` |
| `@RequestBody` | 请求体（JSON） | `{"title": "百度"}` |

### 4.5 Optional 的使用

虽然本项目没有直接使用 `Optional`，但在实际开发中，`Optional` 经常用来优雅地处理"可能为 null"的情况：

```java
// 不用 Optional（容易忘记判空，导致 NullPointerException）
Bookmark bookmark = bookmarkMapper.selectById(id);
if (bookmark != null) {
    return toResponse(bookmark);
} else {
    throw new BusinessException(ErrorCode.BOOKMARK_NOT_FOUND);
}

// 用 Optional（强制你处理 null 情况）
Optional<Bookmark> optional = Optional.ofNullable(bookmarkMapper.selectById(id));
return optional.map(this::toResponse)
        .orElseThrow(() -> new BusinessException(ErrorCode.BOOKMARK_NOT_FOUND));
```

`Optional` 就像一个盒子，里面可能有值，也可能是空的。它的 `map`、`orElseThrow` 等方法帮你安全地处理这两种情况。

本项目选择直接判空的方式，对初学者更友好。但了解 `Optional` 有助于阅读其他项目的代码。
