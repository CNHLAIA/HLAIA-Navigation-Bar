## 2. 概念讲解

### 2.1 什么是树形结构？——用身边的东西类比

**类比 1：公司组织架构**

```
CEO
├── 技术总监
│   ├── 前端组长
│   └── 后端组长
├── 产品总监
└── 运营总监
```

- CEO 是"根节点"——最顶层的那个人
- 技术总监、产品总监是 CEO 的"子节点"
- 前端组长是技术总监的子节点
- 每个人只有一个直属上级（一个 parent），但可以有多个下属（多个 children）
- 这就是一棵**树**

**类比 2：电脑里的文件系统**

```
C:\
├── Users
│   └── Alice
│       ├── Desktop
│       └── Documents
├── Program Files
└── Windows
```

文件系统就是一个经典的树形结构：文件夹里面有子文件夹，子文件夹里面还可以有子文件夹……

**类比 3：我们的书签导航栏**

```
工作资料 (id=1)
├── 前端 (id=2)
│   └── Vue (id=5)
├── 后端 (id=3)
└── 工具 (id=4)
学习资源 (id=6)
├── Java (id=7)
└── Python (id=8)
```

每个文件夹可以有子文件夹，形成无限层级的嵌套——这就是本项目要实现的树形结构。

### 2.2 树形结构的关键术语

| 术语 | 含义 | 项目中的例子 |
|------|------|-------------|
| **节点（Node）** | 树中的每一个元素 | 每一个文件夹就是一个节点 |
| **根节点（Root）** | 最顶层的节点，没有父节点 | "工作资料"、"学习资源"都是根节点 |
| **子节点（Child）** | 某个节点的下一级 | "前端"是"工作资料"的子节点 |
| **父节点（Parent）** | 某个节点的上一级 | "工作资料"是"前端"的父节点 |
| **叶子节点（Leaf）** | 没有子节点的节点 | "Vue"、"工具"、"后端"都是叶子节点 |
| **深度（Depth）** | 从根到该节点的层级数 | "Vue"的深度是 3（根→前端→Vue） |

注意：一棵树可以有**多个根节点**。在我们的项目中，用户可以在顶级目录下创建多个文件夹，每个都是根节点。

### 2.3 难题：数据库怎么存储树？

数据库的表是"扁平"的——每一行就是一条记录，没有"嵌套"。那怎么用扁平的行来表示树形层级关系呢？

答案是：**自引用（Self-Reference）**——在表中加一个字段指向同一张表中的另一行。

看看我们项目的 `folder` 表是怎么设计的：

```sql
CREATE TABLE IF NOT EXISTS `folder` (
    `id`         BIGINT       NOT NULL AUTO_INCREMENT,
    `user_id`    BIGINT       NOT NULL,
    `parent_id`  BIGINT       DEFAULT NULL COMMENT 'NULL means root level',
    `name`       VARCHAR(100) NOT NULL,
    `sort_order` INT          NOT NULL DEFAULT 0,
    `icon`       VARCHAR(50)  DEFAULT NULL,
    `created_at` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at` DATETIME     NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    KEY `idx_user_id` (`user_id`),
    KEY `idx_parent_id` (`parent_id`),
    CONSTRAINT `fk_folder_user` FOREIGN KEY (`user_id`) REFERENCES `user` (`id`) ON DELETE CASCADE,
    CONSTRAINT `fk_folder_parent` FOREIGN KEY (`parent_id`) REFERENCES `folder` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

关键看 `parent_id` 这一行：

- `parent_id BIGINT DEFAULT NULL` —— 这个字段存储的是**父文件夹的 ID**
- `DEFAULT NULL` —— 如果没有父文件夹（即它是顶级文件夹），值为 NULL
- `CONSTRAINT fk_folder_parent FOREIGN KEY (parent_id) REFERENCES folder(id)` —— 外键约束，`parent_id` 引用的是**同一张表的 `id` 字段**，这就是"自引用"

#### 用实际数据来理解

假设我们有这样的树：

```
工作资料 (id=1)
├── 前端 (id=2)
│   └── Vue (id=5)
├── 后端 (id=3)
└── 工具 (id=4)
```

数据库中是这样存储的（扁平的行）：

| id | user_id | parent_id | name | sort_order |
|----|---------|-----------|------|------------|
| 1  | 1       | **NULL**  | 工作资料 | 0 |
| 2  | 1       | **1**     | 前端 | 0 |
| 3  | 1       | **1**     | 后端 | 1 |
| 4  | 1       | **1**     | 工具 | 2 |
| 5  | 1       | **2**     | Vue | 0 |

看到了吗？每一行只记录"我的父节点是谁"。通过 `parent_id` 的指向关系，就能还原出完整的树。

- `parent_id = NULL` 的是根节点（工作资料）
- `parent_id = 1` 的是"工作资料"的子节点（前端、后端、工具）
- `parent_id = 2` 的是"前端"的子节点（Vue）

#### 为什么还有两个索引？

```sql
KEY `idx_user_id` (`user_id`),
KEY `idx_parent_id` (`parent_id`),
```

- `idx_user_id` —— 因为几乎所有查询都要按用户过滤（`WHERE user_id = ?`），加索引让查询更快
- `idx_parent_id` —— 因为构建树和查询子文件夹时都要按 `parent_id` 查找（`WHERE parent_id = ?`），加索引同样是为了性能

> **什么是索引？** 索引就像书的目录——没有目录时你要逐页翻找，有目录你可以直接翻到对应页。数据库索引也是同理，让查询不用扫描全表。

### 2.4 邻接表模型——以及为什么选它

用 `parent_id` 字段存储树形关系，这种方案叫做**邻接表模型（Adjacency List）**。

这是最经典的树形存储方案，但不是唯一的。简单了解下其他方案：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **邻接表** | 每行存 `parent_id` | 简单直观，增删改容易 | 查多层子节点需要递归或多次查询 |
| 路径枚举 | 每行存完整路径如 `"1/2/5"` | 查祖先很简单 | 移动节点要更新所有后代的路径 |
| 嵌套集 | 用左右值标记节点区间 | 查子树快 | 增删改非常复杂，要更新大量节点 |
| 闭包表 | 单独一张表存所有祖先-后代关系 | 查询最灵活 | 存储空间大，维护成本高 |

对于我们的项目来说，**邻接表是最佳选择**，原因：

1. **简单直观**——只需要一个 `parent_id` 字段，初学者也能理解
2. **适合项目规模**——用户的书签文件夹不会太多（通常几十到几百个），递归查询完全够用
3. **增删改容易**——创建文件夹只需设 `parent_id`，移动文件夹只需改 `parent_id`
4. **MyBatis-Plus 配合好**——在 Service 层用 Java 代码递归构建树，不需要写复杂的 SQL

### 2.5 如何从扁平数据构建树？——两种方法

#### 方法一：递归法

从根节点开始，对每个根节点查找它的子节点，再对每个子节点查找子节点的子节点……这就是递归。

```
findChildren(根节点 A)
  ├── 找到子节点 B → findChildren(B)
  │     └── 找到子节点 D → findChildren(D)
  │           └── 没有子节点，结束
  └── 找到子节点 C → findChildren(C)
        └── 没有子节点，结束
```

每次递归都要查一次数据库，如果树有 N 层深，就要查 N 次数据库。当树很深、节点很多时，性能不太好。

#### 方法二：两次遍历法（本项目使用）

**核心思想：先把所有数据一次性查出来，再用 Java 代码在内存中组装。**

第一步：查询该用户的所有文件夹（一条 SQL，一次数据库查询）

```sql
SELECT * FROM folder WHERE user_id = 1 ORDER BY sort_order ASC
```

第二步：把所有文件夹转成 Map（`Map<文件夹ID, 树节点>`），方便按 ID 快速查找

第三步：遍历所有文件夹，通过 `parent_id` 把子节点挂到父节点的 `children` 列表中

```
数据库数据（扁平）：              构建后的树形结构：
id=1, parent_id=NULL           工作资料 (id=1)
id=2, parent_id=1              ├── 前端 (id=2)
id=3, parent_id=1              ├── 后端 (id=3)
id=4, parent_id=1              └── 工具 (id=4)
```

为什么用 Map 不用双层 for 循环？

- 双层 for 循环：对每个文件夹遍历所有文件夹找父节点，时间复杂度 O(n^2)
- 用 Map：通过 ID 直接找到父节点，时间复杂度 O(1)，总复杂度 O(n)

当文件夹数量较多时，Map 方案快得多。

### 2.6 CRUD 操作在树形结构中的特殊之处

| 操作 | 扁平表 | 树形结构 |
|------|--------|---------|
| **创建** | 插入一行即可 | 插入一行，但要设好 `parent_id` 和 `sort_order` |
| **查询** | 查一行或列表 | 要把扁平数据组装成树 |
| **更新（改名）** | 改 name 字段 | 和扁平表一样，没有特殊处理 |
| **删除** | 删一行即可 | 要考虑子文件夹怎么办？级联删除还是拒绝？ |
| **移动** | 不存在这个概念 | 改 `parent_id` 即可，但要防止循环引用 |
| **排序** | 改 sort 字段 | 改 `sort_order` 字段，同级排序 |

重点讲两个容易出错的地方：

#### 移动操作：防止循环引用

假设有树 `A -> B -> C`（A 是 B 的父级，B 是 C 的父级）。如果把 A 移动到 C 下面：

```
移动前：     A → B → C          移动后：   A → B → C → A → B → C → ...
             ^                              ↑___________________|
           根节点                            形成了循环！
```

循环引用会导致：
- 前端渲染无限循环（页面崩溃）
- 后端递归构建树时栈溢出
- 数据完整性被破坏

所以移动前必须检查：**目标位置不能是被移动文件夹的后代**。

#### 删除操作：级联 vs 孤立

删除一个有子文件夹的文件夹时，子文件夹怎么办？

- **级联删除**：连子文件夹一起删掉。我们的 SQL 用了 `ON DELETE CASCADE`，数据库层面会自动级联删除
- **拒绝删除**：如果文件夹有子文件夹就不让删，要求用户先清空
- **移到上级**：把子文件夹移到被删文件夹的父级

本项目选择了数据库级的级联删除（`ON DELETE CASCADE`），简单直接。

### 2.7 排序（sort_order）——拖拽排序的原理

用户在前端拖拽文件夹改变顺序时，发生了什么？

```
拖拽前：                拖拽后（把"工具"拖到最前面）：
前端 (sort=0)           工具 (sort=0)
后端 (sort=1)           前端 (sort=1)
工具 (sort=2)           后端 (sort=2)
```

前端重新计算所有受影响文件夹的 `sort_order`，然后把新的排序列表发给后端：

```json
{
  "items": [
    {"id": 4, "sortOrder": 0},
    {"id": 2, "sortOrder": 1},
    {"id": 3, "sortOrder": 2}
  ]
}
```

后端遍历列表，逐个更新 `sort_order` 字段。

## 3. 代码逐行解读

现在我们来看项目中是怎么实现这些概念的。按照**数据流方向**阅读：Entity → DTO → Mapper → Service → Controller。

### 3.1 Folder 实体类

> 文件：`src/main/java/com/hlaia/entity/Folder.java`

```java
@Data
@TableName("folder")
public class Folder {

    @TableId(type = IdType.AUTO)
    private Long id;

    private Long userId;

    private Long parentId;      // <-- 核心字段：指向父文件夹

    private String name;

    private Integer sortOrder;  // <-- 排序序号

    private String icon;

    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
}
```

**逐字段解释：**

| 字段 | 作用 | 为什么需要 |
|------|------|-----------|
| `id` | 主键，唯一标识一个文件夹 | 所有操作都需要通过 ID 定位文件夹 |
| `userId` | 所属用户 | 每个用户有独立的文件夹树，数据隔离 |
| `parentId` | 父文件夹的 ID | 实现树形结构的核心，NULL 表示根节点 |
| `name` | 文件夹名称 | 用户看到的名字 |
| `sortOrder` | 排序序号，越小越靠前 | 实现自定义排序 |
| `icon` | 图标 | 美化显示 |
| `createdAt` / `updatedAt` | 时间戳 | 审计追踪 |

**核心注解：**

- `@TableName("folder")` —— 告诉 MyBatis-Plus 这个类对应数据库的 `folder` 表
- `@TableId(type = IdType.AUTO)` —— 主键自增策略，数据库自动分配 ID
- `@Data` —— Lombok 注解，自动生成 getter、setter、toString 等方法

### 3.2 DTO 类

#### FolderCreateRequest —— 创建文件夹时前端传什么

> 文件：`src/main/java/com/hlaia/dto/request/FolderCreateRequest.java`

```java
@Data
public class FolderCreateRequest {
    private Long parentId;    // 可选，null 表示根目录

    @NotBlank(message = "文件夹名称不能为空")
    @Size(max = 100, message = "文件夹名称不能超过100个字符")
    private String name;      // 必填

    private String icon;      // 可选
}
```

注意字段分两类：

- **必填**（带校验注解）：`name`
- **可选**（无校验注解）：`parentId`、`icon`

`parentId` 为什么可选？因为 `null` 是合法值——表示在根目录创建文件夹。如果加了 `@NotNull`，就无法表达"在根目录创建"了。

#### FolderSortRequest —— 拖拽排序时前端传什么

> 文件：`src/main/java/com/hlaia/dto/request/FolderSortRequest.java`

```java
@Data
public class FolderSortRequest {

    @NotNull(message = "排序列表不能为空")
    @Valid                    // <-- 重要：让内部 SortItem 的校验注解也生效
    private List<SortItem> items;

    @Data
    public static class SortItem {     // <-- 静态内部类
        @NotNull(message = "文件夹ID不能为空")
        private Long id;

        @NotNull(message = "排序序号不能为空")
        private Integer sortOrder;
    }
}
```

**两个知识点：**

1. **`@Valid` 嵌套校验**——默认情况下，`@NotNull` 只能校验 `items` 本身不为 null，无法校验列表中每个 `SortItem` 的字段。加上 `@Valid`，校验会"穿透"到列表内部。
2. **静态内部类**——`SortItem` 用 `static` 修饰，意味着它不需要先创建 `FolderSortRequest` 对象才能创建。它只是逻辑上属于 `FolderSortRequest`，让代码更有组织性。

#### FolderMoveRequest —— 移动文件夹时前端传什么

> 文件：`src/main/java/com/hlaia/dto/request/FolderMoveRequest.java`

```java
@Data
public class FolderMoveRequest {
    private Long parentId;   // null = 移到根目录，其他值 = 移到指定文件夹下
}
```

只有一个字段。文件夹本身的 ID 通过 URL 路径传递（如 `PUT /api/folders/{id}/move`），不需要放在请求体中。

#### FolderTreeResponse —— 返回给前端的树形结构

> 文件：`src/main/java/com/hlaia/dto/response/FolderTreeResponse.java`

```java
@Data
public class FolderTreeResponse {
    private Long id;
    private Long parentId;
    private String name;
    private String icon;
    private Integer sortOrder;
    private List<FolderTreeResponse> children;    // <-- 核心：递归结构
    private Integer childFolderCount;
    private Integer bookmarkCount;
    private LocalDateTime createdAt;
}
```

**最关键的字段是 `children`：**

```java
private List<FolderTreeResponse> children;
```

这行代码的含义是：一个 `FolderTreeResponse` 对象里，包含一个 `FolderTreeResponse` 列表。每个列表元素又有自己的 `children`……形成无限嵌套。

这种"自己包含自己的列表"的设计叫做**自引用数据结构**，在树形、链表等数据结构中非常常见。

另外注意 `childFolderCount` 和 `bookmarkCount` 是**冗余计算字段**——数据库里没有存这些值，是 Service 层构建树时计算出来的，方便前端显示角标（如 "前端 (5)" 表示 5 个书签）。

### 3.3 FolderMapper

> 文件：`src/main/java/com/hlaia/mapper/FolderMapper.java`

}

### 3.4 FolderService —— 核心业务逻辑

> 文件：`src/main/java/com/hlaia/service/FolderService.java`

这是本课最重要的文件。我们逐方法分析。

#### 类头和依赖注入

```java
@Service
@RequiredArgsConstructor
public class FolderService {

    private final FolderMapper folderMapper;
    private final BookmarkMapper bookmarkMapper;
```

- `@Service` —— 标记为业务逻辑类，Spring 自动创建实例
- `@RequiredArgsConstructor` —— Lombok 自动为 `final` 字段生成构造方法，Spring 通过构造方法注入依赖
- `final` —— 表示赋值后不可修改，确保依赖不会被意外替换

#### getFolderTree() —— 构建 folder 树

这是整个文件夹模块最核心的方法。它把数据库中的扁平数据组装成前端需要的树形结构。

**整体流程：**

```
数据库（扁平行）     →    Java 内存（Map）    →    树形结构
id=1, parent=null         map: {1→node1,         工作资料 (node1)
id=2, parent=1                  2→node2,         ├── 前端 (node2)
id=3, parent=1                  3→node3}         └── 后端 (node3)
```

**第一步：查询该用户的所有文件夹**

```java
List<Folder> allFolders = folderMapper.selectList(
        new LambdaQueryWrapper<Folder>()
                .eq(Folder::getUserId, userId)
                .orderByAsc(Folder::getSortOrder));
```

等价于 SQL：

```sql
SELECT * FROM folder WHERE user_id = ? ORDER BY sort_order ASC
```

一条 SQL 把用户的所有文件夹全查出来，已按 `sort_order` 排好序。

**第二步：转为 Map**

```java
Map<Long, FolderTreeResponse> map = allFolders.stream()
        .collect(Collectors.toMap(Folder::getId, this::toTreeResponse));
```

这段代码用 Stream API 把 `List<Folder>` 转成 `Map<Long, FolderTreeResponse>`：

- `Folder::getId` —— Map 的 key（文件夹 ID）
- `this::toTreeResponse` —— Map 的 value（把 Folder 转成 FolderTreeResponse）
- `this::` 是方法引用的写法，`this::toTreeResponse` 等价于 `folder -> this.toTreeResponse(folder)`

转换后的 Map：

```
{1 → {id:1, name:"工作资料", children:null},
 2 → {id:2, name:"前端", children:null},
 3 → {id:3, name:"后端", children:null},
 4 → {id:4, name:"工具", children:null},
 5 → {id:5, name:"Vue", children:null}}
```

**第三步：统计每个文件夹的书签数量**

```java
List<Bookmark> allBookmarks = bookmarkMapper.selectList(
        new LambdaQueryWrapper<Bookmark>().eq(Bookmark::getUserId, userId));
Map<Long, Long> bookmarkCounts = allBookmarks.stream()
        .collect(Collectors.groupingBy(Bookmark::getFolderId, Collectors.counting()));
```

先查所有书签，再按 `folderId` 分组计数：

- `groupingBy(Bookmark::getFolderId, ...)` —— 按 folderId 分组
- `counting()` —— 每组计数

结果如 `{1: 5, 2: 3}` 表示文件夹 1 有 5 个书签，文件夹 2 有 3 个。

**第四步：构建树形结构（核心）**

```java
List<FolderTreeResponse> roots = new ArrayList<>();
for (Folder folder : allFolders) {
    FolderTreeResponse node = map.get(folder.getId());
    node.setBookmarkCount(bookmarkCounts.getOrDefault(folder.getId(), 0L).intValue());

    if (folder.getParentId() == null) {
        // 根节点 → 加入 roots 列表
        roots.add(node);
    } else {
        // 子节点 → 挂到父节点的 children 中
        FolderTreeResponse parent = map.get(folder.getParentId());
        if (parent != null) {
            if (parent.getChildren() == null) {
                parent.setChildren(new ArrayList<>());  // 懒初始化
            }
            parent.getChildren().add(node);
        }
    }
}
```

让我们用具体数据走一遍这个循环：

```
遍历到 id=1 (工作资料, parentId=null)：
  → parentId 为 null，是根节点
  → roots = [{id:1, name:"工作资料"}]

遍历到 id=2 (前端, parentId=1)：
  → 找到 parent = map.get(1) = "工作资料" 节点
  → 把 "前端" 加入 "工作资料" 的 children
  → roots 不变，但 "工作资料" 现在有了 children

遍历到 id=3 (后端, parentId=1)：
  → 找到 parent = "工作资料" 节点
  → 把 "后端" 加入 "工作资料" 的 children

遍历到 id=4 (工具, parentId=1)：
  → 找到 parent = "工作资料" 节点
  → 把 "工具" 加入 "工作资料" 的 children

遍历到 id=5 (Vue, parentId=2)：
  → 找到 parent = map.get(2) = "前端" 节点
  → 把 "Vue" 加入 "前端" 的 children
```

最终结果：

```json
[
  {
    "id": 1, "name": "工作资料", "parentId": null,
    "children": [
      { "id": 2, "name": "前端", "parentId": 1,
        "children": [
          { "id": 5, "name": "Vue", "parentId": 2, "children": null }
        ]
      },
      { "id": 3, "name": "后端", "parentId": 1, "children": null },
      { "id": 4, "name": "工具", "parentId": 1, "children": null }
    ]
  }
]
```

**懒初始化（Lazy Initialization）：**

```java
if (parent.getChildren() == null) {
    parent.setChildren(new ArrayList<>());
}
```

只有当需要添加子节点时才创建 `children` 列表。没有子文件夹的节点，`children` 就是 `null`。这样节省内存，但前端遍历时需要注意判空。

**第五步：计算子文件夹数量**

```java
for (FolderTreeResponse node : map.values()) {
    if (node.getChildren() != null) {
        node.setChildFolderCount(node.getChildren().size());
    } else {
        node.setChildFolderCount(0);
    }
}
```

遍历 Map 中所有节点，统计 `children` 列表的大小。这一步必须在树形结构构建完成后才能做。

#### createFolder() —— 创建文件夹

```java
@Transactional
public FolderTreeResponse createFolder(Long userId, FolderCreateRequest request) {
    Folder folder = new Folder();
    folder.setUserId(userId);
    folder.setParentId(request.getParentId());
    folder.setName(request.getName());
    folder.setIcon(request.getIcon());

    Long count = folderMapper.selectCount(
            new LambdaQueryWrapper<Folder>()
                    .eq(Folder::getUserId, userId)
                    .eq(request.getParentId() != null, Folder::getParentId, request.getParentId())
                    .isNull(request.getParentId() == null, Folder::getParentId));
    folder.setSortOrder(count.intValue());

    folderMapper.insert(folder);
    return toTreeResponse(folder);
}
```

**条件拼接查询：**

```java
.eq(request.getParentId() != null, Folder::getParentId, request.getParentId())
.isNull(request.getParentId() == null, Folder::getParentId)
```

第一个参数是 `boolean` 条件：
- 当 `request.getParentId() != null` 为 true 时，拼接 `AND parent_id = ?`
- 当 `request.getParentId() == null` 为 true 时，拼接 `AND parent_id IS NULL`

这样一条代码就能处理两种情况，不需要写 `if-else` 分支。

**sortOrder 的设计：** 新建文件夹的 `sortOrder` 设为当前同级文件夹的数量。如果同级已有 3 个文件夹（sortOrder 为 0、1、2），新建的 sortOrder 就是 3，自然排在最后。

#### updateFolder() —— 更新文件夹

```java
@Transactional
public FolderTreeResponse updateFolder(Long userId, Long folderId, FolderCreateRequest request) {
    Folder folder = getFolderForUser(userId, folderId);
    if (request.getName() != null) folder.setName(request.getName());
    if (request.getIcon() != null) folder.setIcon(request.getIcon());
    folderMapper.updateById(folder);
    return toTreeResponse(folder);
}
```

**部分更新策略：** 只更新 `request` 中非 null 的字段。用户只想改名称就只传 `name`，只想改图标就只传 `icon`。不传的字段保持原值不变。

**权限校验：** `getFolderForUser(userId, folderId)` 确保文件夹存在且属于当前用户。

#### deleteFolder() —— 删除文件夹

```java
@Transactional
public void deleteFolder(Long userId, Long folderId) {
    getFolderForUser(userId, folderId);
    folderMapper.deleteById(folderId);
}
```

看起来很简单——先校验权限，再删除。但数据库的 `ON DELETE CASCADE` 外键约束会自动级联删除子文件夹和它们下面的书签。

#### sortFolders() —— 批量排序

```java
@Transactional
public void sortFolders(Long userId, FolderSortRequest request) {
    for (FolderSortRequest.SortItem item : request.getItems()) {
        Folder folder = getFolderForUser(userId, item.getId());
        folder.setSortOrder(item.getSortOrder());
        folderMapper.updateById(folder);
    }
}
```

遍历前端传来的排序列表，逐个更新 `sortOrder`。每次更新前都做权限校验。

为什么需要 `@Transactional`？因为涉及多次 UPDATE 操作，必须保证要么全部成功，要么全部回滚。如果更新到一半失败，排序会处于不一致状态。

#### moveFolder() —— 移动文件夹

```java
@Transactional
public void moveFolder(Long userId, Long folderId, FolderMoveRequest request) {
    Folder folder = getFolderForUser(userId, folderId);
    if (request.getParentId() != null) {
        if (isDescendant(folderId, request.getParentId(), userId)) {
            throw new BusinessException(400, "Cannot move folder into its own descendant");
        }
    }
    folder.setParentId(request.getParentId());
    folderMapper.updateById(folder);
}
```

移动的本质：**改 `parentId` 就够了**。

但移动前要检查防止循环引用——如果目标文件夹是被移动文件夹的后代，就不允许移动。

#### isDescendant() —— 递归判断后代关系

```java
private boolean isDescendant(Long ancestorId, Long candidateId, Long userId) {
    if (ancestorId.equals(candidateId)) return true;      // 终止条件
    List<Folder> children = folderMapper.selectList(
            new LambdaQueryWrapper<Folder>()
                    .eq(Folder::getUserId, userId)
                    .eq(Folder::getParentId, ancestorId));
    for (Folder child : children) {
        if (isDescendant(child.getId(), candidateId, userId)) return true;  // 递归调用
    }
    return false;
}
```

这是本项目中最典型的**递归方法**。让我们用具体例子走一遍：

```
问题：id=1 是否是 id=5 的祖先？（即：把文件夹 1 移到文件夹 5 下会不会形成循环？）

树形结构：1 → 2 → 5

调用 isDescendant(1, 5, userId)
  ├─ 1 == 5? 否，继续
  ├─ 查询 id=1 的子节点，找到 [id=2]
  └─ 递归调用 isDescendant(2, 5, userId)
       ├─ 2 == 5? 否，继续
       ├─ 查询 id=2 的子节点，找到 [id=5]
       └─ 递归调用 isDescendant(5, 5, userId)
            ├─ 5 == 5? 是！返回 true
```

因为 id=5 确实是 id=1 的后代，所以不能把 id=1 移到 id=5 下面。

递归的两个要素：
1. **终止条件**：`ancestorId.equals(candidateId)` —— 如果找到循环了就停止
2. **递归步骤**：对每个子节点调用自身 —— 一层层往下找

#### getFolderForUser() —— 权限校验辅助方法

```java
private Folder getFolderForUser(Long userId, Long folderId) {
    Folder folder = folderMapper.selectById(folderId);
    if (folder == null || !folder.getUserId().equals(userId)) {
        throw new BusinessException(ErrorCode.FOLDER_NOT_FOUND);
    }
    return folder;
}
```

无论文件夹不存在还是不属于当前用户，都抛出"文件夹不存在"异常。为什么不区分？

- 安全考虑：如果返回"无权限"，攻击者就知道这个 ID 对应的资源确实存在
- 返回"不存在"，攻击者无法判断是资源不存在还是没有权限
- 这种做法叫做**模糊响应**，是安全的最佳实践

#### toTreeResponse() —— Entity 转 DTO

```java
private FolderTreeResponse toTreeResponse(Folder folder) {
    FolderTreeResponse dto = new FolderTreeResponse();
    dto.setId(folder.getId());
    dto.setParentId(folder.getParentId());
    dto.setName(folder.getName());
    dto.setIcon(folder.getIcon());
    dto.setSortOrder(folder.getSortOrder());
    dto.setCreatedAt(folder.getCreatedAt());
    return dto;
}
```

把数据库实体转为前端需要的 DTO。注意 `children`、`childFolderCount`、`bookmarkCount` 没有在这里设置——它们是在 `getFolderTree()` 方法中后续计算和填充的。

### 3.5 FolderController —— RESTful API

> 文件：`src/main/java/com/hlaia/controller/FolderController.java`

Controller 层不包含业务逻辑，只负责接收请求、调用 Service、返回结果。

**API 一览：**

| HTTP 方法 | URL | 功能 | Controller 方法 |
|-----------|-----|------|----------------|
| GET | `/api/folders/tree` | 获取文件夹树 | `getTree()` |
| POST | `/api/folders` | 创建文件夹 | `create()` |
| PUT | `/api/folders/{id}` | 更新文件夹 | `update()` |
| DELETE | `/api/folders/{id}` | 删除文件夹 | `delete()` |
| PUT | `/api/folders/sort` | 批量排序 | `sort()` |
| PUT | `/api/folders/{id}/move` | 移动文件夹 | `move()` |

**RESTful 设计原则回顾：**

- GET 用于查询——不修改数据，安全且幂等
- POST 用于创建——每次调用可能产生不同结果
- PUT 用于更新——调用一次和多次效果相同（幂等）
- DELETE 用于删除——也是幂等的

**获取用户 ID 的方式：**

```java
@GetMapping("/tree")
public Result<List<FolderTreeResponse>> getTree(@AuthenticationPrincipal Long userId) {
    return Result.success(folderService.getFolderTree(userId));
}
```

`@AuthenticationPrincipal Long userId` 直接从 Spring Security 上下文中取出当前登录用户的 ID。这是在 JWT 过滤器中放入的，不需要前端额外传递——既方便又安全。

**路径变量 vs 请求体：**

```java
@PutMapping("/{id}")
public Result<FolderTreeResponse> update(@AuthenticationPrincipal Long userId,
                                          @PathVariable Long id,
                                          @Valid @RequestBody FolderCreateRequest request) {
```

- `@PathVariable Long id` —— 从 URL 中取（如 `/api/folders/42` → `id = 42`）
- `@RequestBody FolderCreateRequest request` —— 从请求体 JSON 中取
- `@Valid` —— 触发 DTO 上的校验注解

**为什么移动操作用 `PUT /{id}/move` 而不是简单的 `PUT /{id}`？**

因为普通的 `PUT /{id}` 已经被"更新文件夹信息"（改名、改图标）占用了。移动是一个不同的操作，需要额外的路径 `/move` 来区分。这种把操作名放在路径末尾的方式叫做**RESTful 子资源操作**。

## 4. 关键 Java 语法点

### 4.1 递归方法

递归是指**方法调用自身**。在树形结构中非常常见。

```java
// 判断 candidateId 是否是 ancestorId 的后代
private boolean isDescendant(Long ancestorId, Long candidateId, Long userId) {
    // 终止条件（基准情况）：找到了循环
    if (ancestorId.equals(candidateId)) return true;
    // 递归步骤：对每个子节点继续判断
    List<Folder> children = folderMapper.selectList(...);
    for (Folder child : children) {
        if (isDescendant(child.getId(), candidateId, userId)) return true;
    }
    return false;
}
```

**递归三要素：**

1. **终止条件**——必须有，否则无限循环导致 `StackOverflowError`。这里终止条件是 `ancestorId.equals(candidateId)`
2. **递归调用**——方法调用自身，但参数要向终止条件靠近。这里每次递归往下找一层子节点
3. **返回值**——把子调用的结果传递给父调用

**递归的内存模型：**

每次递归调用都会在栈（Stack）上创建一个新的方法帧。如果树太深（比如 1000 层），可能耗尽栈空间导致 `StackOverflowError`。但文件夹树通常不会太深（3-5 层已经很极限了），所以递归完全安全。

### 4.2 Stream API 的常用操作

本项目大量使用了 Java Stream API。以下是关键用法：

**collect(Collectors.toMap())** —— 收集为 Map

```java
// 把 List<Folder> 转成 Map<文件夹ID, 树节点>
Map<Long, FolderTreeResponse> map = allFolders.stream()
        .collect(Collectors.toMap(Folder::getId, this::toTreeResponse));
```

- `Folder::getId` —— Map 的 key（方法引用，等价于 `folder -> folder.getId()`）
- `this::toTreeResponse` —— Map 的 value（把每个 Folder 转成 DTO）

**collect(Collectors.groupingBy())** —— 分组

```java
// 按 folderId 分组并计数
Map<Long, Long> bookmarkCounts = allBookmarks.stream()
        .collect(Collectors.groupingBy(Bookmark::getFolderId, Collectors.counting()));
```

- `groupingBy(Bookmark::getFolderId, ...)` —— 按 folderId 分组
- `counting()` —— 每组计数
- 结果：`{1: 5, 2: 3}` 表示文件夹 1 有 5 个书签

**map.getOrDefault()** —— 安全取值

```java
bookmarkCounts.getOrDefault(folder.getId(), 0L)
```

如果 Map 中有这个 key 就返回对应值，没有就返回默认值 `0L`。避免了手动判断 null。

### 4.3 List 和嵌套 List（自引用结构）

```java
public class FolderTreeResponse {
    private List<FolderTreeResponse> children;  // 自己包含自己的列表
}
```

这个看似奇怪的声明完全合法。它表达的是：一个文件夹树节点包含一个子节点列表，每个子节点又是一个文件夹树节点。

在 JSON 中表现为：

```json
{
  "id": 1, "name": "工作资料",
  "children": [
    { "id": 2, "name": "前端",
      "children": [
        { "id": 5, "name": "Vue", "children": null }
      ]
    }
  ]
}
```

嵌套可以无限深入，完全由数据决定。

### 4.4 条件拼接查询

```java
new LambdaQueryWrapper<Folder>()
    .eq(Folder::getUserId, userId)                                          // 始终生效
    .eq(request.getParentId() != null, Folder::getParentId, request.getParentId())  // 条件生效
    .isNull(request.getParentId() == null, Folder::getParentId)             // 条件生效
```

MyBatis-Plus 的 `LambdaQueryWrapper` 支持条件拼接：

- `.eq(条件, 字段, 值)` —— 第一个参数为 `true` 时才拼接这个条件
- `.isNull(条件, 字段)` —— 同理

这比写 `if-else` 分支拼接 SQL 要简洁得多。

### 4.5 @Transactional 注解

```java
@Transactional
public void sortFolders(Long userId, FolderSortRequest request) {
    for (FolderSortRequest.SortItem item : request.getItems()) {
        Folder folder = getFolderForUser(userId, item.getId());
        folder.setSortOrder(item.getSortOrder());
        folderMapper.updateById(folder);
    }
}
```

`@Transactional` 的作用：保证方法内的所有数据库操作**要么全部成功，要么全部失败（回滚）**。

如果排序更新到第 3 个文件夹时抛出异常：
- 没有 `@Transactional`：前 2 个已更新，第 3 个没更新，数据不一致
- 有 `@Transactional`：前 2 个的更新也会回滚，数据保持一致

**什么时候需要加？**

| 场景 | 是否需要 @Transactional |
|------|------------------------|
| 单个查询 | 不需要 |
| 单个插入 | 可以不加（一条 SQL 本身是原子的） |
| 多个更新 | **需要** |
| 先查后改 | **需要**（防止并发问题） |
| 涉及多张表的操作 | **需要** |

## 小结

本课我们学习了：

1. **树形结构**的概念——用公司组织架构和文件系统做类比
2. **邻接表模型**——用 `parent_id` 字段在数据库中存储树形关系
3. **两次遍历法**——先查所有数据建 Map，再遍历挂载 children，构建树形结构
4. **递归**——方法调用自身，用于判断后代关系
5. **CRUD 操作**在树形结构中的特殊处理——创建时设 parentId，移动时防循环，排序时用 sortOrder
6. **Stream API** 的实际应用——`Collectors.toMap`、`Collectors.groupingBy`
7. **@Transactional**——保证多个数据库操作的原子性

文件夹树形结构是后端开发中非常常见的需求。理解了邻接表模型和两次遍历法，你就能应对大多数树形数据场景了。
