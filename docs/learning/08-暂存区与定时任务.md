### 1.2 时间处理基础 —— LocalDateTime

在之前的课程中，我们见过 `LocalDateTime` 类型的 `createdAt` 和 `updatedAt` 字段。这节课的核心字段 `expireAt`（过期时间）也是这个类型。我们先来系统了解一下它。

`LocalDateTime` 是 Java 8 引入的时间类，表示"一个不带时区的日期 + 时间"。你可以把它理解为"2024年6月15日 下午3点30分"这样的精确时刻。

```java
import java.time.LocalDateTime;

// 获取当前时间
LocalDateTime now = LocalDateTime.now();
// 输出示例：2024-06-15T15:30:00.123456789

// 时间计算：在当前时间基础上加上一段时间
LocalDateTime tomorrow = now.plusDays(1);       // 加1天
LocalDateTime oneHourLater = now.plusHours(1);  // 加1小时
LocalDateTime later = now.plusMinutes(1440);    // 加1440分钟（= 24小时 = 1天）

// 时间比较：判断一个时间是否在另一个时间之前/之后
boolean isBefore = now.isBefore(tomorrow);    // true（现在比明天早）
boolean isAfter = now.isAfter(oneHourLater);  // false（现在比1小时后早）
```

这三组方法——**获取当前时间**、**时间计算**、**时间比较**——就是暂存区过期时间管理所需的全部工具。

### 1.3 什么是定时任务（Scheduled Task）？

你一定用过手机上的闹钟：设定一个时间，到了就响。定时任务的概念完全一样——**让程序在特定的时间间隔或时间点自动执行某段代码**。

生活中的定时任务例子：
- 手机闹钟：每天早上 7 点响
- 空调定时关机：2 小时后自动关闭
- 邮件定时发送：每天早上 9 点发送日报

在本项目中，定时任务用来**自动清理过期的暂存区数据**。就像你需要定期清理桌面上的过期文件一样，程序也需要定期清理数据库中"过了保质期"的记录。

Spring Boot 中使用定时任务只需要两步：
1. 在主类上加 `@EnableScheduling` 注解（开闹钟功能）
2. 在方法上加 `@Scheduled` 注解（设定闹钟时间和要做的事）

我们稍后会详细讲解。

## 2. 概念讲解

### 2.1 暂存区（Staging Area）是什么？

#### 一个生活中的类比

想象你有一个**书桌**和一个**书架**：

| | 书桌（暂存区） | 书架（正式书签） |
|---|---|---|
| 使用方式 | 随手往桌上一扔就行 | 需要精心摆放，分门别类 |
| 归类要求 | 没有，就是平摊着 | 必须放到某个分类格里 |
| 保质期 | 东西放久了会被清理 | 永久保存 |
| 编辑 | 基本不改 | 可以随时修改描述、排序 |
| 使用场景 | "先放着，回头再整理" | "已经整理好的收藏" |

暂存区就是这个"书桌"——用户浏览网页时，通过浏览器扩展一键把网页"扔"到桌上，不需要选文件夹、不需要填描述。之后有空了，再打开网页端，慢慢把书桌上的东西归置到书架上对应的位置。

#### 暂存区 vs 书签的对比

| 特性 | 暂存区（Staging） | 书签（Bookmark） |
|---|---|---|
| 保存方式 | 浏览器扩展一键保存 | 网页端手动创建 |
| 文件夹归属 | 无（扁平列表） | 必须属于某个文件夹 |
| 过期时间 | 有（默认1天） | 无（永久保存） |
| 可编辑字段 | 仅过期时间 | 标题、链接、描述 |
| 排序方式 | 按创建时间倒序 | 支持手动排序 |
| 数据库表 | `staging_item` | `bookmark` |

#### 暂存区的使用流程

```
用户浏览网页
    |
    v
浏览器扩展一键保存 -----> 暂存区（staging_item 表）
    |                        |
    | 不需要选文件夹          | 有过期时间（默认1天）
    | 不需要填描述            | 过期后自动清理
    |
    v
用户打开网页端查看暂存区
    |
    +---> 感兴趣的网页 --> "移到文件夹" --> 变成正式书签（bookmark 表）
    |
    +---> 不需要的网页 --> 手动删除
    |
    +---> 没处理的网页 --> 过期后自动清理
```

### 2.2 过期时间管理：expireAt 字段

#### 设计思路

每个暂存项都有一个 `expireAt` 字段，表示它的"保质期截止时间"。这是一个 `LocalDateTime` 类型的值，记录的是一个精确的未来时刻，比如 `2024-06-16T15:30:00`。

```
创建暂存项的时间线：

|---- 创建时间 ----|-------------------------|---- 过期时间 ----|
                  ^                                          ^
                  |                                          |
            LocalDateTime.now()              LocalDateTime.now().plusMinutes(1440)
                  |                                          |
            2024-06-15 15:30                2024-06-16 15:30（+1天）
```

#### 创建时如何计算过期时间

过期时间的计算公式非常简单：**当前时间 + 过期分钟数 = 过期时间**。

```java
// 默认：1天后过期（1440 分钟 = 24 小时 × 60 分钟）
LocalDateTime.now().plusMinutes(1440);

// 1小时后过期
LocalDateTime.now().plusMinutes(60);

// 7天后过期（7 × 24 × 60 = 10080 分钟）
LocalDateTime.now().plusMinutes(10080);
```

#### 默认过期时间的选择

项目选择了 1440 分钟（1天）作为默认值。这个选择有讲究：

- **太短**（如 1 小时）：用户可能来不及查看和整理，暂存项就过期消失了，失去意义
- **太长**（如 7 天）：暂存区会堆积大量内容，用户看到一堆东西会焦虑，失去了"临时"的定位
- **1 天**：给用户足够的时间整理，又不会堆积太久——一个合理的平衡点

用户也可以在保存时自定义过期时间：
- `expireMinutes = 60`：1 小时后过期（适合"看完就扔"的临时链接）
- `expireMinutes = 1440`：1 天后过期（默认值，大多数场景）
- `expireMinutes = 10080`：7 天后过期（适合"周末慢慢看"的长文）

#### 查询时如何过滤过期数据

查询暂存区列表时，只返回还没有过期的暂存项：

```java
// SQL 思路：WHERE expire_at > 当前时间
.gt(StagingItem::getExpireAt, LocalDateTime.now())
```

`.gt()` 是 "greater than" 的缩写，表示 `expire_at > NOW()`，即过期时间在当前时间之后——也就是"还没过期的"。

#### 更新过期时间

用户可以延长暂存项的"保质期"。注意，更新时是**以当前时间为基准重新计算**，而不是在原过期时间上加减：

```
原过期时间：2024-06-16 15:30
当前时间：2024-06-16 10:00
用户设置：expireMinutes = 60

新的过期时间 = 当前时间 + 60分钟 = 2024-06-16 11:00
（而不是 原过期时间 + 60分钟 = 2024-06-16 16:30）
```

这种设计更直觉——"从现在起再给我 1 小时"，而不是"在原来的基础上再加 1 小时"。

### 2.3 @Scheduled 定时任务详解

#### 为什么需要定时任务？

当暂存项过期后，虽然查询时会被过滤掉（用户看不到），但过期的数据仍然躺在数据库里，占用空间。我们需要一种机制**自动清理**这些过期数据。

这就是定时任务的用武之地——每隔一段时间，程序自动扫描数据库，找出所有 `expire_at <= 当前时间` 的记录，然后删除它们。

#### fixedRate vs fixedDelay vs cron

Spring 的 `@Scheduled` 注解提供了三种方式来指定执行频率：

| 方式 | 含义 | 示例 | 适用场景 |
|---|---|---|---|
| `fixedRate` | 固定频率：从上一次**开始**执行时计时 | `@Scheduled(fixedRate = 60000)` | 周期性检查（不管上次执行多久） |
| `fixedDelay` | 固定延迟：从上一次**结束**时计时 | `@Scheduled(fixedDelay = 60000)` | 上次执行完等 N 毫秒再执行 |
| `cron` | Cron 表达式：指定精确时间点 | `@Scheduled(cron = "0 0 2 * * ?")` | 在特定时间点执行 |

三者的执行时间线对比：

```
fixedRate = 60000（固定频率）：
|--执行10s--|----等待50s----|--执行10s--|----等待50s--|
^           ^60s                         ^120s

fixedDelay = 60000（固定延迟）：
|--执行10s--|------等待60s------|--执行15s--|------等待60s------|
^                             ^70s                           ^145s

cron = "0 * * * * ?"（每分钟整点）：
:00秒  :60秒  :120秒  :180秒
  |       |       |       |
  v       v       v       v
 执行    执行    执行    执行
```

本项目使用 `fixedRate = 60000`（每 60 秒扫描一次），因为清理过期暂存项是一个"持续监控"的任务，需要频繁检查。

#### Cron 表达式基础

Cron 表达式是一种时间表达语法，格式为 6 个字段（Spring 的格式）：

```
┌─── 秒（0-59）
│ ┌─── 分（0-59）
│ │ ┌─── 时（0-23）
│ │ │ ┌─── 日（1-31）
│ │ │ │ ┌─── 月（1-12）
│ │ │ │ │ ┌─── 星期（0-7，0和7都是周日）
│ │ │ │ │ │
* * * * * *
```

常用示例：

```
"0 0 2 * * ?"      每天凌晨 2 点整
"0 0/5 * * * ?"    每 5 分钟执行一次（0分、5分、10分...）
"0 0 * * * ?"      每小时整点执行
"0 0 9-17 * * ?"   每天 9 点到 17 点，每小时执行
"0 0 0 1 * ?"      每月 1 号零点执行
```

特殊字符含义：
- `*`：任意值（每个时间单位都匹配）
- `?`：不指定值（只能用在"日"和"星期"字段，表示"不关心"）
- `/`：步进值（如 `0/5` 表示从 0 开始每 5 单位）
- `-`：范围（如 `9-17` 表示 9 到 17）

#### @EnableScheduling 的作用

`@EnableScheduling` 是一个总开关。它告诉 Spring："请创建一个后台线程池，专门用来执行 `@Scheduled` 注解标记的方法。"

如果忘记添加这个注解：
- 代码不会报错（`@Scheduled` 注解不会导致编译错误或运行时异常）
- 但定时任务永远不会执行（这是新手最容易踩的坑！）

本项目中，它加在主类 `HlaiaNavigationBarApplication` 上：

```java
@SpringBootApplication
@EnableScheduling  // 开启定时任务支持，少了这行，所有 @Scheduled 方法都不会执行
public class HlaiaNavigationBarApplication {
    public static void main(String[] args) {
        SpringApplication.run(HlaiaNavigationBarApplication.class, args);
    }
}
```

### 2.4 "移入文件夹"操作：暂存区到书签的转换

这是暂存区最核心的操作——将一个临时暂存项"转正"为永久书签。整个过程分三步：

```
暂存项数据：                         书签数据：
┌──────────────────────┐            ┌──────────────────────────┐
│ title: "Vue3 官网"    │            │ title: "Vue3 官网"        │
│ url: "https://vue..." │  转换      │ url: "https://vue..."     │
│ iconUrl: "..."        │ ========> │ iconUrl: "..."            │
│ userId: 1             │            │ userId: 1                 │
│ （无 folderId）       │            │ folderId: 3  ← 新增！      │
│ （无 sortOrder）      │            │ sortOrder: 5 ← 新增！      │
│ expireAt: 2024-...   │            │ （无 expireAt，永久保存）    │
└──────────────────────┘            └──────────────────────────┘
```

关键点：
1. **数据复制**：把暂存项的 `title`、`url`、`iconUrl` 复制到新的 `Bookmark` 对象
2. **补充归属**：设置 `folderId`（用户指定的目标文件夹）
3. **删除原暂存项**：从暂存区移除（因为已经"转正"了）

这三步必须是一个**原子操作**（要么全成功，要么全失败），由 `@Transactional` 保证。

### 3.1 Entity 层：StagingItem.java

> 文件路径：`src/main/java/com/hlaia/entity/StagingItem.java`

```java
@Data
@TableName("staging_item")   // 对应数据库的 staging_item 表
public class StagingItem {

    @TableId(type = IdType.AUTO)
    private Long id;

    private Long userId;         // 所属用户ID

    private String title;        // 网页标题（浏览器扩展自动获取）
    private String url;          // 网页链接地址
    private String iconUrl;      // 网站图标 URL

    private LocalDateTime expireAt;   // ★ 核心：过期时间
    private LocalDateTime createdAt;  // 创建时间（保存到暂存区的时间）
}
```

和之前学过的 `Bookmark` 实体对比一下：

| StagingItem 有 | Bookmark 有 | 说明 |
|---|---|---|
| `expireAt` | 无 | 暂存项有过期时间，书签永久保存 |
| 无 `folderId` | `folderId` | 暂存项不需要归属文件夹 |
| 无 `description` | `description` | 暂存项不需要描述 |
| 无 `sortOrder` | `sortOrder` | 暂存项按时间排序，不需要手动排序 |

可以看到，`StagingItem` 比 `Bookmark` 精简很多。这体现了**"暂存区追求快、书签追求精细"**的设计理念。

`expireAt` 是本阶段最重要的字段。它存储的是一个精确的未来时刻（如 `2024-06-16T15:30:00`），超过这个时间后，定时任务会自动清理这条记录。

### 3.2 DTO 层

#### StagingCreateRequest.java

> 文件路径：`src/main/java/com/hlaia/dto/request/StagingCreateRequest.java`

```java
@Data
public class StagingCreateRequest {

    @NotBlank(message = "网页标题不能为空")

    @NotBlank(message = "网页链接不能为空")

    private Integer expireMinutes;  // 过期时间（分钟），可选
```

注意 `expireMinutes` 的类型是 `Integer`（包装类型）而不是 `int`（基本类型）。这是因为：

```java
// int 的默认值是 0 —— 无法区分"用户没传"和"用户传了0"
int a;  // 默认值 = 0

// Integer 的默认值是 null —— 可以明确区分
Integer b;  // 默认值 = null
// b == null → 用户没传，使用默认值 1440
// b == 60   → 用户传了 60 分钟
```

`title` 和 `url` 是必填的，因为保存一个网页至少需要这两个信息。`expireMinutes` 是可选的，不传就使用默认值。

### 3.4 Service 层：StagingService.java

> 文件路径：`src/main/java/com/hlaia/service/StagingService.java`

这是本阶段的重点文件。我们逐方法解读。

#### 依赖注入

```java
@Service
@RequiredArgsConstructor
public class StagingService {

    private final StagingItemMapper stagingItemMapper;  // 暂存项 Mapper
    private final BookmarkMapper bookmarkMapper;         // 书签 Mapper（moveToFolder 需要用到）
```

`StagingService` 需要两个 Mapper：
- `StagingItemMapper`：操作暂存项表（增删改查）
- `BookmarkMapper`：操作书签表（"移入文件夹"时需要创建书签）

#### addToStaging() —— 添加到暂存区

```java
@Transactional
public StagingItemResponse addToStaging(Long userId, StagingCreateRequest request) {
    // 第一步：构建 Entity 对象
    StagingItem item = new StagingItem();
    item.setUserId(userId);
    item.setTitle(request.getTitle());
    item.setUrl(request.getUrl());

    // 第二步：计算过期时间
    int expireMinutes = request.getExpireMinutes() != null ? request.getExpireMinutes() : 1440;
    item.setExpireAt(LocalDateTime.now().plusMinutes(expireMinutes));

    // 第三步：插入数据库
    stagingItemMapper.insert(item);
    return toResponse(item);
}
```

逐行解读：

**第一步**：把 DTO 的数据复制到 Entity。这在前几课已经见过很多次了。

**第二步**的核心是这一行：

```java
int expireMinutes = request.getExpireMinutes() != null ? request.getExpireMinutes() : 1440;
```

这是**三元运算符**，格式是 `条件 ? 真值 : 假值`：
- 如果用户传了 `expireMinutes`（不为 null），就用用户传的值
- 如果没传（为 null），就用默认值 1440（1天）

然后：

```java
item.setExpireAt(LocalDateTime.now().plusMinutes(expireMinutes));
```

`LocalDateTime.now()` 获取当前时间，`.plusMinutes(1440)` 加上 1440 分钟（1天），结果就是一个未来时刻。例如现在是 `2024-06-15 15:30`，过期时间就是 `2024-06-16 15:30`。

**第三步**：调用 `insert` 插入数据库，然后通过 `toResponse` 把 Entity 转为 DTO 返回。

#### getStagingItems() —— 查询暂存区列表

```java
public List<StagingItemResponse> getStagingItems(Long userId) {
    List<StagingItem> items = stagingItemMapper.selectList(
            new LambdaQueryWrapper<StagingItem>()
                    .eq(StagingItem::getUserId, userId)             // 只查当前用户的
                    .gt(StagingItem::getExpireAt, LocalDateTime.now()) // 过滤已过期项
                    .orderByDesc(StagingItem::getCreatedAt));        // 最新的排最前
    return items.stream().map(this::toResponse).collect(Collectors.toList());
}
```

这个查询构建了三个条件：

1. `.eq(StagingItem::getUserId, userId)` —— 数据隔离，只查当前用户的暂存项
2. `.gt(StagingItem::getExpireAt, LocalDateTime.now())` —— 只返回还没过期的（`expire_at > 当前时间`）
3. `.orderByDesc(StagingItem::getCreatedAt)` —— 按创建时间倒序，最新的排在最前面

等价的 SQL：
```sql
SELECT * FROM staging_item
WHERE user_id = ? AND expire_at > NOW()
ORDER BY created_at DESC
```

为什么在数据库层面过滤过期项，而不是查出全部再在 Java 中过滤？因为数据库有索引优化，效率更高。如果用户有 500 个暂存项（大部分已过期），全部查出来再过滤会浪费大量内存和网络带宽。

最后的 `stream().map(this::toResponse).collect(Collectors.toList())` 是 Java Stream API 的标准用法：对列表中的每个元素调用 `toResponse` 方法进行转换。

#### updateExpiry() —— 修改过期时间

```java
@Transactional
public StagingItemResponse updateExpiry(Long userId, Long itemId, StagingUpdateRequest request) {
    StagingItem item = getStagingItemForUser(userId, itemId);  // 校验权限
    item.setExpireAt(LocalDateTime.now().plusMinutes(request.getExpireMinutes()));
    stagingItemMapper.updateById(item);
    return toResponse(item);
}
```

逻辑很简单：
1. 先通过 `getStagingItemForUser` 校验暂存项是否存在且属于当前用户
2. 以当前时间为基准重新计算过期时间
3. 更新数据库

注意过期时间的重新计算方式：`LocalDateTime.now().plusMinutes(...)`。这意味着新的过期时间是**从现在开始算**，而不是在原有过期时间上加减。

#### deleteStagingItem() —— 删除暂存项

```java

}
```

用户主动删除不需要的暂存项。注意这里是**物理删除**（直接从数据库中移除记录），因为暂存区本身就是临时数据，没有恢复的必要。

#### moveToFolder() —— 移入文件夹（核心操作）

```java
@Transactional
public void moveToFolder(Long userId, Long itemId, MoveToFolderRequest request) {
    // 第一步：校验暂存项
    StagingItem item = getStagingItemForUser(userId, itemId);

    // 第二步：从暂存项创建书签
    Bookmark bookmark = new Bookmark();
    bookmark.setUserId(userId);
    bookmark.setFolderId(request.getFolderId());  // 用户指定的目标文件夹
    bookmark.setTitle(item.getTitle());            // 复制标题
    bookmark.setUrl(item.getUrl());                // 复制链接
    bookmark.setIconUrl(item.getIconUrl());        // 复制图标

    // 计算排序序号：新书签排在文件夹的最后面
    Long count = bookmarkMapper.selectCount(
            new LambdaQueryWrapper<Bookmark>()
                    .eq(Bookmark::getFolderId, request.getFolderId()));
    bookmark.setSortOrder(count.intValue());

    bookmarkMapper.insert(bookmark);

    // 第三步：删除暂存项
    stagingItemMapper.deleteById(itemId);
}
```

这是暂存区最核心的方法，需要重点关注：

**第一步**：校验暂存项是否存在且属于当前用户。这和之前所有 Service 中的权限校验模式一样。

**第二步**：创建书签。把暂存项的 `title`、`url`、`iconUrl` 复制到新的 `Bookmark` 对象中，然后设置用户指定的 `folderId`。

`sortOrder` 的计算方式：统计目标文件夹下已有多少书签，新书签的 `sortOrder` 就是已有数量（排在最后）。例如文件夹下已有 3 个书签（`sortOrder` 为 0、1、2），新书签的 `sortOrder` 就是 3。

**第三步**：删除暂存项。

**为什么必须加 `@Transactional`？** 因为这个方法涉及两次写操作（一次 `insert` + 一次 `delete`），它们必须在同一个事务中：
- 如果 `insert` 成功但 `delete` 失败 → 暂存区里还留着一个"幽灵记录"，用户可能再次操作，导致重复创建书签
- 如果 `delete` 成功但 `insert` 失败 → 用户的数据丢失了！暂存项没了，书签也没创建

`@Transactional` 保证了"要么全成功，要么全失败"，不会出现中间状态。

#### getStagingItemForUser() —— 权限校验辅助方法

```java
private StagingItem getStagingItemForUser(Long userId, Long itemId) {
    StagingItem item = stagingItemMapper.selectById(itemId);
    if (item == null || !item.getUserId().equals(userId)) {
        throw new BusinessException(ErrorCode.STAGING_NOT_FOUND);
    }
    return item;
}
```

这和之前 `BookmarkService` 中的权限校验方法模式完全一样：
1. 根据主键 ID 查询实体
2. 判断 `userId` 是否匹配
3. 不匹配时抛出"不存在"异常（而不是"无权限"异常）

为什么返回"不存在"而不是"无权限"？这是安全最佳实践——如果返回"无权限"，攻击者就知道这个 ID 对应的资源确实存在；返回"不存在"，攻击者无法判断是资源不存在还是没有权限。

### 3.6 定时任务：StagingCleanupScheduler.java

> 文件路径：`src/main/java/com/hlaia/scheduled/StagingCleanupScheduler.java`

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class StagingCleanupScheduler {

    private final StagingItemMapper stagingItemMapper;
    private final KafkaProducer kafkaProducer;

    @Scheduled(fixedRate = 60000)  // 每 60 秒执行一次
    public void scanExpiredItems() {
        // 第1步：查询所有过期的暂存项
        List<StagingItem> expired = stagingItemMapper.selectList(
                new LambdaQueryWrapper<StagingItem>()
                        .le(StagingItem::getExpireAt, LocalDateTime.now()));

        // 第2步：遍历每条过期记录，通过 Kafka 发送清理消息
        for (StagingItem item : expired) {
            kafkaProducer.sendStagingCleanup(item.getId(), item.getUserId());
        }

        // 第3步：记录日志
        if (!expired.isEmpty()) {
            log.info("Scheduled cleanup: {} expired staging items", expired.size());
        }
    }
}
```

逐段解读：

**`@Scheduled(fixedRate = 60000)`**：每 60000 毫秒（60秒）执行一次。从上一次任务**开始执行时**计时，不管任务执行多久。

**第1步：查询过期暂存项**

```java
.le(StagingItem::getExpireAt, LocalDateTime.now())
```

`.le()` 是 "less than or equal"（小于等于），等价于 SQL：
```sql
SELECT * FROM staging_item WHERE expire_at <= NOW()
```

找出所有"过期时间已经过去"的暂存项。注意这和查询列表时的 `.gt()` 正好相反：
- 查询列表：`.gt()`（大于当前时间 = 还没过期的）
- 清理任务：`.le()`（小于等于当前时间 = 已经过期的）

**第2步：通过 Kafka 发送清理消息**

项目采用了一个架构设计：定时任务**只负责扫描（发现过期数据），不负责删除**。实际的删除操作通过 Kafka 消息队列异步执行。

为什么扫描和删除要分开？

1. **单一职责**：扫描器的职责是"发现需要清理的数据"，消费者的职责是"执行清理操作"
2. **可靠性**：如果直接删除失败，没有重试机制；通过 Kafka 传递，如果消费者处理失败，Kafka 会重新投递消息
3. **可扩展性**：如果未来删除逻辑变复杂（如需要先备份），只需修改消费者，扫描器不用动
4. **负载均衡**：如果部署了多个服务实例，Kafka 会自动分配消息给不同实例

**第3步：记录日志**

```java
if (!expired.isEmpty()) {
    log.info("Scheduled cleanup: {} expired staging items", expired.size());
}
```

只在有过期数据时才输出日志。如果没有过期数据（大部分情况），就不输出——避免每分钟都刷一条无用的日志，污染日志文件。

**`@Component` 注解**：告诉 Spring "这是一个组件，请创建它的实例"。和 `@Service` 的作用基本一样，只是语义上 `@Component` 更通用，`@Service` 更明确表示"业务逻辑类"。定时任务类用 `@Component` 即可。

**`@Slf4j` 注解**：Lombok 自动生成 `log` 对象，之后可以直接用 `log.info()`、`log.error()` 等方法输出日志。等价于手写：
```java
private static final Logger log = LoggerFactory.getLogger(StagingCleanupScheduler.class);
```

## 4. 关键 Java 语法点

### 4.1 LocalDateTime 和时间计算

`LocalDateTime` 是 `java.time` 包中的类，Java 8 引入，用来替代老旧的 `Date` 和 `Calendar`。

本项目用到的方法总结：

| 方法 | 作用 | 示例 |
|---|---|---|
| `LocalDateTime.now()` | 获取当前时间 | `2024-06-15T15:30:00` |
| `.plusMinutes(n)` | 加 n 分钟 | `now.plusMinutes(1440)` → 明天此刻 |
| `.plusHours(n)` | 加 n 小时 | `now.plusHours(1)` → 1 小时后 |
| `.plusDays(n)` | 加 n 天 | `now.plusDays(7)` → 7 天后 |
| `.isBefore(other)` | 当前时间是否在 other 之前 | `now.isBefore(tomorrow)` → true |
| `.isAfter(other)` | 当前时间是否在 other 之后 | `now.isAfter(yesterday)` → true |

与之对应的还有减法方法：`.minusMinutes()`、`.minusHours()`、`.minusDays()`。

注意 `LocalDateTime` 是**不可变的**（immutable）——每次调用 `plusMinutes` 等方法都会返回一个**新对象**，不会修改原来的对象：

```java
LocalDateTime now = LocalDateTime.now();        // 2024-06-15 15:30
LocalDateTime tomorrow = now.plusDays(1);       // 2024-06-16 15:30

// now 仍然是 2024-06-15 15:30，没有被修改
// tomorrow 是一个新对象
```

### 4.2 @Value 注解读取配置

虽然本项目的定时任务没有使用 `@Value`，但这是一个和 `@Scheduled` 配合使用的常见注解，值得了解。

`@Value` 可以从 `application.yml` 中读取配置值，用在 `@Scheduled` 中可以实现"可配置的执行频率"：

```java
// 在 application.yml 中配置：
// staging.cleanup.interval: 60000

@Scheduled(fixedRate = 60000)                    // 硬编码，改频率需要改代码重新编译
public void scanExpiredItems() { ... }

@Scheduled(fixedRateString = "${staging.cleanup.interval}")  // 从配置文件读取，改频率只需改配置
public void scanExpiredItems() { ... }
```

当使用 `@Value` 从配置中读取值时，要用 `fixedRateString`（字符串版本）而不是 `fixedRate`（整数版本）。

### 4.3 @Scheduled 和 cron 表达式

三种使用方式的完整对比：

```java
// 方式一：fixedRate —— 固定频率（最常用）
@Scheduled(fixedRate = 60000)          // 每 60 秒执行一次
public void task1() { ... }

// 方式二：fixedDelay —— 固定延迟（适合执行时间不固定的任务）
@Scheduled(fixedDelay = 60000)         // 上次执行完后等 60 秒再执行
public void task2() { ... }

// 方式三：cron —— 精确时间点（适合在特定时间执行的任务）
@Scheduled(cron = "0 0 2 * * ?")       // 每天凌晨 2 点执行
public void task3() { ... }
```

项目选择 `fixedRate` 的原因：清理过期暂存项需要持续监控，不需要精确到某个时间点，每分钟检查一次就足够了。

Cron 表达式速查表：

```
字段位置：  秒  分  时  日  月  星期
范围：     0-59 0-59 0-23 1-31 1-12 0-7

常用模式：
"0 0 * * * ?"        每小时整点
"0 */5 * * * ?"      每 5 分钟
"0 0 2 * * ?"        每天凌晨 2 点
"0 0 0 1 * ?"        每月 1 号零点
"0 0 9-17 * * ?"     每天 9 到 17 点，每小时
"0 0 9-17 * * MON-FRI"  工作日 9 到 17 点，每小时
```

### 4.4 事务管理：moveToFolder 中同时操作两个表

`moveToFolder` 方法是本项目中事务管理的典型案例——它同时操作两张表：

```java
@Transactional
public void moveToFolder(Long userId, Long itemId, MoveToFolderRequest request) {
    // ...校验暂存项...

    bookmarkMapper.insert(bookmark);      // 操作 1：往 bookmark 表插入记录
    stagingItemMapper.deleteById(itemId);  // 操作 2：从 staging_item 表删除记录
}
```

`@Transactional` 的工作原理：

```
方法开始 → 开启事务
    |
    +---> 操作1：INSERT INTO bookmark ...  （还没真正写入数据库）
    |
    +---> 操作2：DELETE FROM staging_item ...  （还没真正删除）
    |
方法正常结束 → 提交事务（两步操作同时生效）
                    |
                    +---> INSERT 生效
                    +---> DELETE 生效

如果中间任何一步抛出异常：
方法异常结束 → 回滚事务（两步操作都不生效）
                    |
                    +---> INSERT 被撤销
                    +---> DELETE 被撤销
```

没有 `@Transactional` 的后果：
- `insert` 成功，`delete` 失败 → 暂存项还在，书签也创建了（重复了）
- `insert` 失败，`delete` 成功 → 暂存项没了，书签也没了（数据丢失）

这就是为什么涉及多步写操作的方法必须加 `@Transactional`。

### 4.5 LambdaQueryWrapper 中的条件方法

本阶段用到的查询条件方法总结：

| 方法 | 含义 | SQL 等价 | 本项目中的使用 |
|---|---|---|---|
| `.eq()` | 等于 | `WHERE x = ?` | `.eq(StagingItem::getUserId, userId)` |
| `.gt()` | 大于 | `WHERE x > ?` | `.gt(StagingItem::getExpireAt, now)` 查询未过期的 |
| `.le()` | 小于等于 | `WHERE x <= ?` | `.le(StagingItem::getExpireAt, now)` 查询已过期的 |
| `.orderByDesc()` | 降序排序 | `ORDER BY x DESC` | `.orderByDesc(StagingItem::getCreatedAt)` |

其中 `.gt()` 和 `.le()` 是一对相反的操作，分别用于"查询未过期的暂存项"和"查询已过期的暂存项"。

### 练习 3：思考题——为什么 moveToFolder 不直接在 StagingItem 表加 folderId？

暂存区和书签分开两张表设计，而不是在 `staging_item` 表上增加 `folderId` 字段（有值就是书签，没值就是暂存项）。想想看，这种设计有什么好处？

<details>
<summary>点击查看思路</summary>

1. **职责分离**：暂存项和书签的字段差异很大（暂存项有 `expireAt`，书签有 `sortOrder`、`description`），放在一张表会有大量空字段
2. **查询效率**：查暂存项不需要关心 `folderId`、`sortOrder` 等字段；查书签不需要过滤 `expireAt`
3. **索引优化**：每张表可以针对自己的查询模式建立索引，互不影响
4. **代码清晰**：两个 Entity、两个 Mapper、两个 Service，各管各的，代码更易理解
</details>

### 练习 4：阅读代码，回答问题

打开 `StagingService.java`，找到 `addToStaging` 方法，回答以下问题：

1. 如果前端不传 `expireMinutes`，暂存项的过期时间是多久？
2. 如果前端传了 `expireMinutes = 120`，过期时间是怎么计算的？
3. 为什么用 `Integer` 而不是 `int` 作为 `expireMinutes` 的类型？

<details>
<summary>点击查看答案</summary>

1. 默认 1440 分钟 = 24 小时 = 1 天
2. `LocalDateTime.now().plusMinutes(120)` —— 当前时间 + 2 小时
3. `int` 默认值是 0，无法区分"没传"和"传了 0"；`Integer` 默认值是 null，可以明确区分
</details>

### 练习 5：扩展思考——如何支持"批量移入文件夹"？

当前的 `moveToFolder` 只能一次移动一个暂存项。如果用户想一次选择 5 个暂存项全部移到同一个文件夹，你会怎么设计 API？

<details>
<summary>点击查看思路</summary>

方案一：新建一个批量接口
```java
@PostMapping("/batch-move-to-folder")
public Result<Void> batchMoveToFolder(
        @AuthenticationPrincipal Long userId,
        @RequestBody BatchMoveToFolderRequest request) {
    // request 包含 List<Long> itemIds 和 Long folderId
    stagingService.batchMoveToFolder(userId, request.getItemIds(), request.getFolderId());
    return Result.success();
}
```

方案二：前端循环调用单个接口（简单但不推荐，无法保证原子性）

方案一更优，因为可以在一个事务中完成所有操作。Service 层的实现就是在 `@Transactional` 方法中循环处理每个暂存项。

## 小结

这节课我们学习了：

1. **暂存区的设计理念**：快存慢整，临时存放 vs 永久收藏
2. **过期时间管理**：`expireAt` 字段的设计、计算方式、查询过滤
3. **`@Scheduled` 定时任务**：`fixedRate` / `fixedDelay` / `cron` 三种方式，`@EnableScheduling` 总开关
4. **"移入文件夹"操作**：暂存项到书签的转换，`@Transactional` 保证原子性
5. **时间处理 API**：`LocalDateTime` 的 `now()`、`plusMinutes()`、`isBefore()` 等方法

核心收获：
- 暂存区是"书桌"，书签是"书架"，各有各的定位
- 过期时间 = 当前时间 + 过期分钟数，用 `LocalDateTime` 处理
- 定时任务用 `@Scheduled` + `@EnableScheduling`，别忘了加总开关
- 跨表操作必须加 `@Transactional`，保证原子性
